## Headless / CLI rituals

- **Runtime captures must run WINDOWED, not `-batchmode`.** `-batchmode` produces no real rendered frames; screenshot evidence comes from launching the built exe windowed (`-screen-fullscreen 0`) with an in-game capture component.
  - **Nuance ‚Äî the RT-readback lever (confirmed via the #248 spike 2026-07-03, PENDING MERGE; the runner-unpin path for ticket `86cag93zb`):** the "`-batchmode` produces no frames" rule above is really TWO separable causes: (a) **`-nographics` forces a Null graphics device** ‚Üí nothing renders at all (the spike's `-batchmode -nographics` frame came back variance 0 / dead); (b) even with a device, Unity's backbuffer `ScreenCapture` has no swapchain/backbuffer to read under batchmode (hence the historical windowed `-screen-fullscreen 0` requirement). BUT rendering a camera to an **offscreen `RenderTexture` ‚Üí `Texture2D.ReadPixels` ‚Üí PNG** works under `-batchmode` **WITHOUT `-nographics`**: a real GPU device (D3D12) initializes and the spike frame was non-black + content-correct (frame_check PASS: mean_luma 134.4 / variance 230.6 / magenta 0). So window-less, session-independent headless captures ARE possible via RT-readback + `-batchmode` (no `-nographics`, no `-screen-fullscreen 0`) ‚Äî removing the swapchain/desktop-compositor contention that pins the CI capture job to one runner. ‚öÝ Two caveats before converting the gates: the multi-submesh URP mis-render caution below (editor `Camera.Render` in batchmode, hero-axe #21) still applies to material-FIDELITY judgment captures; and the shipped standalone-PLAYER `-batchmode` device-init (vs the editor-batchmode the spike proved) is NOT yet proven ‚Äî validate a full-build player inits a GPU device offscreen first. **UPDATE (PR #250, pending merge): the player device-init IS proven** ‚Äî shipped exe under `-batchmode` (no `-nographics`) inits `device=Direct3D12`, and the legacy backbuffer `ScreenCapture` is confirmed DEAD headless ("Failed to capture screen shot") ‚Äî RT-readback (via `SubmitRenderRequest`) is the only headless capture path. **Hard boundary found: gates whose evidence includes IMGUI / UI-Toolkit OVERLAYS (settings panel, loot/inventory drag, water-HUD class) CANNOT go headless via a camera RT** ‚Äî screen-space overlays composite to the backbuffer/swapchain, not into a camera's RenderTexture, so those gates stay WINDOWED on the pinned runner; only pure world-camera gates (spawn/chop/held-belt/sky class) convert. **Reviewer-ritual trap (Tess, #287 QA 2026-07-08):** re-running a `-verify*` gate WITH `-nographics` yields `device=Null` + `RenderTexture.Create failed` + identical BLACK captures while the LOGIC gate still exits 0 ‚Äî a false-empty visual that looks like "the feature renders nothing." Reviewers re-running verify gates must DROP `-nographics` to get real frames; judge visuals only from a run with a real device line in the log.
- **Editor automation pattern:** `Unity.exe -batchmode -quit -projectPath <p> -executeMethod <Class.Method> -logFile <log>` for scene/asset work; `-runTests -testPlatform EditMode|PlayMode -testResults <xml>` for tests (NO `-quit` with `-runTests`). Always grep the XML's `<test-run ... result= total= passed= failed=` line ‚Äî exit code alone lies on some failure classes.
- **Headless PlayMode time trap:** `Time.deltaTime ‚âà 0` per frame in headless runs ‚Äî never assert on per-frame deltas; sample over a real `Time.time` window instead. **Sibling trap (PR #290, 2026-07-08): `Time.unscaledDeltaTime`-driven tweens FREEZE headless** ‚Äî a regrow/fade tween stepping by unscaledDeltaTime never advances, so "X never happened" failures can be the tween frozen, not the feature broken. Historical chop-triage postscript: the 8-test over-count was NEVER the clock (`captureDeltaTime` is honored headless on dev box AND CI, probe run 28926857959, confirmed 86camf6vz) ‚Äî it was 3 test-side mechanisms (same-frame re-arm spun by `while(SwingInProgress)` steppers; renderer-captured-before-construction; frozen unscaledDeltaTime tween). Prefer deterministic owned clocks (the #288 `TestClock` pattern) for cadence tests.
  - **Cadence-gate OVER-COUNT + the `Time.captureDeltaTime` remedy (Tess, PR #255 / FH-PMTRIAGE-CHOP `86cajt6j8`).** The `deltaTime‚âà0` trap has a second face beyond "per-frame deltas lie": a test whose logic is gated on time-SPACED cadence steps (e.g. a chop's impact at ~0.1s THEN the clip-completion/next-swing at ~0.3s) OVER-COUNTS headlessly ‚Äî because `deltaTime‚âà0` while `Time.time` leaps coarsely, ONE frame can both resolve an impact AND begin the next swing, so N real swings register as >N (observed 2-vs-1, 4-vs-1, 5-vs-3, 12-vs-5). This looks EXACTLY like a real double-apply / "machine-gun" gameplay bug ‚Äî do NOT blind-quarantine it (that would hide a genuine one). Discriminate: (a) confirm the production path single-flights structurally/clock-independently (increments once behind a guard + resolves ‚â§1 pending-impact per frame with the next step scheduled in the FUTURE), and (b) confirm the shipped windowed `-verify*` capture PASSES at real framerate. If both hold, the reds are the headless clock, not the code. **Fix = pin `Time.captureDeltaTime` (a fixed virtual step, e.g. 0.01f) in the test's SetUp + restore to 0 in TearDown** ‚Äî this gives a deterministic finer-than-60fps clock so cadence windows span MULTIPLE frames again, keeping the tests LIVE-deterministic (a real machine-gun bug still reds them) rather than `[Ignore]`-quarantined (which loses the regression guard). Pin must be SetUp/TearDown-scoped (restore to 0) to avoid cross-class leak.
- **Local `-batchmode -runTests -testPlatform PlayMode` can DEADLOCK at "Entering Playmode" on this machine ‚Äî EditMode is unaffected; CI is the authoritative PlayMode gate (observed by Devon during the `86caac81y` air-control work, 2026-06-17).** Reproducible on a CLEAN tree before any test runs ‚Üí it's machine/environment state, NOT the code. Two gotchas: (1) a `taskkill` of a hung PlayMode run leaves stray `Assets/InitTestScene*.unity` files that RE-HANG the next run ‚Äî delete them before retrying. (2) `serve_soak.sh` does NOT run PlayMode, so soak builds are unaffected. When a local PlayMode run deadlocks, don't fight it ‚Äî run EditMode locally and let the PR's CI run be the authoritative PlayMode result (the self-hosted runner runs the identical command on a fresh checkout and passes).
- **`WaitForEndOfFrame` is NOT evoked in `-batchmode` (Devon, PR #60 locomotion-harness work).** A PlayMode coroutine `yield return new WaitForEndOfFrame()` never resumes headlessly (there is no end-of-frame render pass to fire it), so any value you intended to sample AFTER `LateUpdate`/render is never read ‚Äî the test hangs or silently skips the assertion. To sample a post-`LateUpdate` value in a headless PlayMode coroutine, record it in a high-`DefaultExecutionOrder` `LateUpdate` component and read the PRIOR frame's recorded value on the next `yield return null` ‚Äî never gate the sample on `WaitForEndOfFrame`.
- **A non-completing bootstrap can ship a STALE scene to CI's EditMode gate, false-REDding scene-presence tests (Drew, ticket 86cabtc83 / Unity 6000.4.10f1‚Üí6000.4.11f1 upgrade, PR #103).** The CI `unity` job runs `BootstrapProject.Run` (re-bakes `Boot.unity` from a fresh empty scene) BEFORE EditMode. On the COLD-cache FIRST run of a NEWLY-INSTALLED editor, package resolution can be CANCELLED before `Run()` executes at all ‚Äî the bootstrap log ends in `[Package Manager] Failed to resolve packages: operation cancelled.` (+ `IPCStream (Upm-...): IPC stream failed to read (Not connected)`) with NO `[BootstrapProject] start`/`complete` marker. Yet `bootstrap_with_retry.sh` reported SUCCESS off Unity's ADVISORY `-quit` exit code (it can be 0 even when the editor quit without running the `-executeMethod`), so the scene was NEVER re-baked and EditMode opened the **stale COMMITTED `Boot.unity`** instead. The committed scene had been stale since the binary-scene merges that don't re-commit it (CI re-bakes, so they merged green) ‚Äî so it lacked every component added since (HungerNeed/AxePickup/InventoryUI/CameraFollowNudgeTool, `followLerp=12` pre-tightening) ‚Üí 6 scene-presence tests went RED, looking exactly like a 6000.4.11f1 behavior regression. **It was NOT a regression: the SAME editor's warm-cache bootstrap (the later advisory `playmode` job) authored all components + logged `[BootstrapProject] complete`** ‚Äî proof the bootstrap CODE is editor-version-agnostic; the cold package-resolve was the only flake. **Diagnosis discriminators:** (1) the failing run's bootstrap log lacks `[BootstrapProject] complete`; (2) the missing components EXACTLY match the stale committed scene (`grep -c HungerNeed Assets/Scenes/Boot.unity` = 0); (3) a value reverted to an OLD default (`followLerp` 18‚Üí12) is the tell that the OLD committed component, not a fresh-default one, was loaded; (4) a warm re-bake on the same editor authors everything. **Fix (PR #103):** gate bootstrap success on the `[BootstrapProject] complete` LOG MARKER, never the advisory exit code (a non-completing bootstrap now fails the step LOUD + RED), and broaden the retry signature beyond `EPERM:...rename` to also cover the `Failed to resolve packages: operation cancelled` / UPM-IPC-drop transient. Regression-guarded in `tests/scripts/test_gate_scripts.sh` (the marker-absent cold-cancel case must NOT report success). This is the inverse of the stale-committed-scene note below: there CI re-bakes so CI stays green while LOCAL is red; here the re-bake itself silently no-op'd so CI went red on a stale scene.
- **Run `BootstrapProject.Run` BEFORE any LOCAL EditMode run ‚Äî a bare local `-runTests -testPlatform EditMode` false-REDs ~5 scene-presence tests (Drew, ticket `86cacvhf4` ‚Üí not-repro, 2026-06-23).** The CI `unity` job ALWAYS bootstraps before EditMode (`ci.yml:161‚Üí172`), so CI is green; a bare LOCAL run skips it, and `Boot.unity`'s bootstrap-authored components/assets (`InventoryUI` theme/PanelSettings, `EnsureRuntimeTheme`'s `.tss`, etc.) are never committed ‚Üí the scene-presence asserts hit null ‚Üí ~5 reds that LOOK like a regression but are NOT. Proof: bare EditMode = `5 failed`; after `BootstrapProject.Run` (log marker `[BootstrapProject] complete`) = `440/440 Passed`. **Discriminator before filing a "regression": if the local EditMode reds are all scene-presence / null-component asserts AND the same tests are green on the PR's CI `unity` job, it's the missing-bootstrap artifact ‚Äî run bootstrap first, don't file a bug.** (A test that 'fails' locally but does NOT reproduce on a fresh `origin/main` checkout ‚Äî e.g. `HungerNeed.startFull` ‚Äî is a separate stale-worktree-state artifact, also not a bug.) This is the LOCAL-red / CI-green inverse of the cold-cache CI case above.
- **`Assert.Inconclusive` is a HARD CI RED here ‚Äî NOT a soft skip (Devon's #119 review, 2026-06-23).** The CI green gate requires `inconclusive == 0` (`.github/workflows/scripts/parse_test_results.py`), so ANY test that ends `Inconclusive` FAILS the gate red. The bootstrap-precondition guardrail (`Assets/Tests/EditMode/BootstrapPrecondition.cs`, PR #119) uses this deliberately + SAFELY: it's INERT on the CI path (CI always bootstraps ‚Üí the asserted object is present ‚Üí no Inconclusive ‚Üí CI green) and only converts a BARE-local no-bootstrap red into an actionable hint; but if a real code change DROPS a bootstrap-authored component, the guard fires Inconclusive even post-bootstrap ‚Üí CI red ‚Üí **so it cannot mask a regression** (the masking-safety property that gated the #119 review). Takeaway: never reach for `Assert.Inconclusive` as a "skip this on CI" ‚Äî it REDS the gate. A conditional skip must be gated on a precondition CI never hits (like missing-bootstrap), so the Inconclusive only ever fires on a path CI doesn't take.
- **A hung self-hosted `unity` job surfaces as `cancelled`, not `failure` ‚Äî and may be a TRANSIENT runner wedge that a re-run clears (#101, 2026-06-22).** A self-hosted job that HANGS (no progress) runs until GitHub's hard ~24h max-job-runtime ceiling, then the run's overall conclusion shows `cancelled` (NOT `failure`, and NOT a human cancel). So a `gh run list` entry with a ~24h job duration + `cancelled` is the signature of a HUNG job that hit the ceiling ‚Äî read it as a hang, never as someone cancelling. The hang can be transient runner-env state (the PackageCache-EPERM / HID-USB-freeze class above), NOT a code/build bug: #101's required `unity` job (EditMode+build+capture) ran 24h11m ‚Üí cancelled, then `gh run rerun <id>` on the now-idle/cleared runner passed clean (unity + structure SUCCESS, same SHA `00901a15a`). **Disambiguate before assuming a code bug:** re-run the same run on an idle runner FIRST; pair the wait with a bounded background watchdog (poll `gh run view --json status,jobs` + a cutoff past a healthy unity-job window, ~25‚Äì35 min) so a re-hang can't silently sit another 24h vs. a bare wait. **‚öÝ Run-level `cancelled` is the NORMAL state for these PRs ‚Äî judge by per-JOB conclusions, NOT the run conclusion.** The advisory `playmode` job historically ALWAYS hung at play-mode-enter and was cancelled at its own ~21-min timeout (**UPDATED 2026-07-08: since PR #249 ‚Äî the 2026-07-03 InitTestScene unbounded-unload fix ‚Äî the suite COMPLETES in ~200s and reports real results; the live set is 10 test-defect failures under triage, PR #282; a playmode-driven run-level `cancelled` is now rare, but the judge-per-JOB rule below stands unchanged**), which made the OVERALL run `conclusion` read `cancelled` EVEN WHEN the required `unity` + `structure` jobs SUCCEEDED (directly confirmed on the clean #101 re-run: unity SUCCESS in ~6 min, structure SUCCESS, playmode cancelled at ~21 min ‚Üí run conclusion `cancelled`). So NEVER read a required-gate verdict off the run-level `conclusion` ‚Äî always check the per-JOB conclusions (`gh run view <id> --json jobs`: unity + structure SUCCESS = required gates green; `gh pr checks` also shows the split). The 24h-ceiling case above is the SEPARATE, rarer signature: it's the REQUIRED `unity` job itself showing a ~24h duration (a real hang/wedge to re-run), vs. this routine ~21-min advisory-playmode cancellation.
- **A CI artifact-upload step that runs BEFORE the gate it claims to log produces a STALE-LOG FALSE-GREEN on a `clean:false` warm runner (the #130 pond round-5 false-green, Devon 2026-06-25, run 28155640831).** The `unity` job uploads `ci-out/*.log` at step 5 (`Upload test results + logs`, right after the build) but the shipped-build GATES (capture/settings/pond) run LATER (steps 7-9). With `clean:false` the gitignored `ci-out/` PERSISTS between runs, so step 5's `ci-out/*.log` glob swept up a STALE `verify-pond.log` left by the PREVIOUS run (round-4, pre-annulus-gate). A reviewer read that stale log from the `unity-ci-logs-<sha>` artifact as THIS PR's gate output ‚Äî it showed pre-round-5 code, looking exactly like "the new gate never ran on the shipped exe" (the hypothesis was a stale incremental player assembly). **It was NOT a stale assembly ‚Äî it was a poisoned evidence trail.** Ground-truth refutation chain (do this BEFORE accepting a "stale build" hypothesis): (1) the REAL gate log lives in the gate's OWN post-gate artifact (`FarHorizon-pond-caps-<sha>/verify-pond.log`) ‚Äî it carried the correct new-gate output (`SHORELINE-ANNULUS PASS` + the 6-token complete line); the two same-named logs in the two artifacts DIFFER (`diff -q` proves it). (2) `build.log` showed a FRESH `Csc ‚Ä¶/FarHorizon.Runtime.dll` (+ HEAD-source compiler warnings like `FreshwaterPondVerifyCapture.cs(341,‚Ä¶)`) ‚Üí the player DLL was recompiled, not stale. (3) The shipped managed DLL itself (`Build/Windows/FarHorizon_Data/Managed/FarHorizon.Runtime.dll` from the `FarHorizon-Windows-<sha>` artifact) CONTAINS the new code ‚Äî verify by searching the DLL for the new string literals in **UTF-16LE** (managed `ldstr` literals are wide; an ASCII `grep`/`strings` MISSES them and lies ‚Äî and `strings` may not even be installed on Git-bash, failing silently to 0). A python `data.count(s.encode('utf-16-le'))` is the reliable test; the format-string FRAGMENT ADJACENCY/ORDER (` topNoWhite=` then ` topNoShorelineRing=` then ` sideSunk=`) discriminates new-vs-old code in the DLL. **Fixes (PR #130 round 6):** clear stale gate logs/caps from the warm `ci-out/` at job START (a new early step), and SCOPE the pre-gate upload to only the logs that exist by step 5 (bootstrap/editmode/build/playmode + XMLs) ‚Äî NOT the gate logs, which are uploaded by their own post-gate steps. General rule: an `if: always()` upload glob must not promise to carry a log written by a LATER step; either scope the glob to pre-existing files or move the upload after the producing step. Sibling of the stale-committed-scene CI traps above ‚Äî same family: the artifact a reviewer trusts is not the one the run actually produced.
- **Build verification:** the builder must exit non-zero on failure and log `result=Succeeded size=<bytes>`; grep that line, never trust silence.
- **Build-stamp ritual:** HUD shows `BUILD <tag> | <UTC> | <sha>`; every soak/judgment verifies the stamp first (three-builds-in-play identity confusion is a proven failure mode).
- **Stale-stamp trap (the soak handoff):** `FarHorizonBuilder.BuildWindows` does NOT regenerate `Assets/Resources/BuildStamp.txt` ‚Äî ONLY `BootstrapProject.Run` (`WriteBuildStamp`) writes it, and the stamp is COMMITTED, so it is ALWAYS stale vs a later HEAD. A soak that runs a bare `BuildWindows` ships the old committed sha ‚Üí build-identity ambiguity (bit the first manual desktop soak, 2026-06-12). **Never serve a soak with a bare build.** Use the one entry point `.github/workflows/scripts/serve_soak.sh` (ticket 86ca86gde): from a CLEAN checkout it chains bootstrap (fresh-stamps HEAD) ‚Üí BuildWindows ‚Üí `verify_build_stamp.py` (fails loud unless the shipped stamp's sha == HEAD) ‚Üí `capture_gate.sh` (reuses U7's frame gate) ‚Üí restores the bootstrap's tracked-asset churn (`git checkout -- Assets ProjectSettings`; the clean-tree precondition makes whole-root restore complete + drift-proof ‚Äî bootstrap's churn set drifts as it grows, so an enumerated path list goes stale). Prints the exe path + HUD stamp + capture dir handoff block. Regression-guarded by `verify_build_stamp.py` + the stale-vs-HEAD case in `tests/scripts/test_gate_scripts.sh`.
- **Capture-evidence freshness: cited == attached == HEAD.** Recurring failure class (3√ó on 2026-06-12: PR #10 stale-stamp PNGs, PR #13 capture-not-attached): a Self-Test Report CITES a capture/stamp, but the artifact on disk is from an older sha, or lives only in gitignored `ci-out/`/`Captures/` where the reviewer can't see it. The gate rests on report text alone ‚Äî exactly what the capture gate exists to prevent. Convention: (1) the capture's HUD stamp must read the PR HEAD sha at citation time; (2) include the verify-run's exit-code log line verbatim in the PR; (3) image attachment to PRs is structurally IMPOSSIBLE from agents (gitignore + structure_check.sh gate committed captures; gist upload classifier-denied for private-repo screenshots; gh has no comment-attachment API) ‚Äî the canonical verification is the REVIEWER INDEPENDENTLY RE-RUNNING the verify gate (serve_soak.sh / -verifyChop class) and judging their own artifact (PR #13 precedent, 2026-06-12). Reviewers: always check cited-stamp == artifact-stamp == HEAD before trusting any visual claim. Same class, CI flavor (4th hit, PR #14): "CI green" claims must cite the actual run ID on the PR head ‚Äî local EditMode/PlayMode/serve_soak results are NOT CI; a PR body asserting CI-green while the head's only run is red fails the evidence gate even when the code is approve-quality. Self-hosted-runner note (CONFIRMED 2√ó 2026-06-12, runs 27433014608 + PR #15 first attempt): concurrent Unity instances collide on PackageCache (`EPERM rename` in bootstrap) ‚Üí environmental CI red. Policy: SERIALIZE CI-triggering pushes (don't push two PRs back-to-back while the runner is busy); on a hit, one re-run ‚Äî it clears when the lock holder exits. The SAME collision hits a dispatched QA REVIEWER's independent-reproduction build: their `serve_soak` runs Unity in their worktree on the same self-hosted-runner machine, so it races an active CI build ‚Üí EPERM for one of them. When you dispatch a reviewer who will build while the PR's CI is still running, message them to HOLD the shipped-build step until CI clears (do code review meanwhile), then release them (near-miss 2026-06-13, PR #26: Tess dispatched into a live unity CI run; serialized by message before her build started). The `gh run list` (no-in-flight-CI) check is necessary but NOT sufficient (PR #36): a still-RESIDENT LOCAL Unity/exe process ‚Äî from a just-finished `serve_soak` or local test that hasn't fully exited ‚Äî can hold the PackageCache lock and trigger the same EPERM even when no CI run is in-flight. Let local Unity/exe processes fully EXIT before pushing a CI-triggering commit. Unity incremental builds can leave the old exe mtime unchanged even on a successful rebuild. Always grep the build log for `result=Succeeded` (plus `size=<bytes>`); use the `BurstDebugInformation_DoNotShip/` folder mtime as a secondary freshness check when the log is unavailable.
- **A shipped-exe capture can STILL be FALSE-GREEN if its camera/lighting ‚âÝ gameplay (86ca8ca1m soak, 2026-06-13).** The castaway recolor passed Tess's isolated `-verifyCastaway` shipped capture (a purpose-built hero framing) AND every guard ‚Äî yet the Sponsor's GAMEPLAY orbit-cam view washed the character to a yellow blob (the scene's warm key + bloom blew the bright atlas out; the isolated verify rig didn't reproduce that lighting). Even a real shipped-exe frame lies if the `-verify<X>` mode frames/lights the subject differently than actual play. Lesson: judgment-grade VISUAL captures must use the GAMEPLAY camera (the orbit cam at its real pitch) + the actual scene lighting/post ‚Äî not a flattering isolated hero shot. Reviewers + capture gates should prefer a gameplay-orbit frame; an isolated verify capture is a smoke-test that the asset EXISTS, NOT proof of how it READS in play. (This is the sibling of the editor-vs-shipped false-negative below: same trap, capture-rig flavor.) **To BUILD a trustworthy verify capture (PR #37 recipe):** ride the gameplay render path (post-processing ON + skybox/scene ambient); frame from settled `SkinnedMeshRenderer.BakeMesh` bounds ‚Äî NEVER raw `.bounds` or a `Mathf.Max` height floor (invalid skinned bounds ‚Üí non-deterministic distance ‚Üí cropped/rear shots); pin a KNOWN facing (don't assume +Z=front); use a NEUTRAL clear backdrop (a warm/bright clear blows the subject's R-channel to cream); and UNIT-TEST the framing math (the bug slipped 3 PRs precisely because the framing math was untested). Built this way the capture is deterministic (byte-identical across runs) AND gameplay-representative.
- **A relative `-captureDir` makes capture PNGs vanish SILENTLY (ticket `86caa9zpp`, PR #226, merged 2026-07-02).** A capture entry point that takes `-captureDir <relative-path>` resolves it against the Unity PROCESS's current working directory, not the project root or the exe's location ‚Äî if that CWD differs from what the caller assumes, the PNGs write somewhere unexpected with no error, and the capture step looks like it silently no-op'd. `CaptureGate` normalizes with `Path.GetFullPath` on parse specifically to avoid this. **Rule: any NEW capture entry point must normalize `-captureDir` (and any other output-path CLI arg) with `Path.GetFullPath` at parse time** ‚Äî never trust a relative path handed in from an unknown launch CWD.
- **"Can't see X in the build" is often a CAMERA-reach / occlusion bug, NOT an asset bug (PR #35 ocean soak-fix, 2026-06-13).** The Sponsor's "I see no ocean" + the ticket both blamed the water material ‚Äî but the water shipped CORRECT. Real causes: (a) the orbit-cam pitch FLOOR (minPitch 35) never tilted low enough to frame the sea, and (b) a beach dune crest occluded the water between the locked spawn and the waterline. Before re-authoring an asset that "doesn't render," diagnose-via-trace: where does the gameplay camera actually point (pitch/zoom clamp range)? is any geometry between the camera and the target? Fixing the camera reach + removing the occluder beat re-tuning the material. Pairs with the false-green-capture trap above ‚Äî both are "the asset is fine, the VIEW is the problem."
- **`BootstrapProject.Run` dirties tracked assets.** Running bootstrap dirtied Boot.unity, several materials, GraphicsSettings.asset, and BuildStamp.txt in observed runs. Always follow bootstrap with `git checkout -- Assets/ ProjectSettings/` before committing or opening PRs ‚Äî bootstrap churn must not ship in unrelated work.
- **`git checkout -- Assets/ ProjectSettings/` ALSO reverts UNCOMMITTED source `.cs` edits under `Assets/Scripts/` (Devon, 2026-06-14).** That restore is a blanket discard of every tracked file under `Assets/` ‚Äî including your in-flight `.cs` source ‚Äî so running bootstrap + restore mid-task silently wipes any source edits you haven't committed (Devon lost all 5 mid-task edits this way; caught only because a verify-grep returned 0). Safe pattern for any worktree that mixes source edits with a bootstrap regenerate: **(1) COMMIT the `.cs` work first, (2) run `BootstrapProject.Run` headless, (3) restore with `git checkout -- Assets/ ProjectSettings/` + surgically re-add only `Assets/Scenes/Boot.unity Assets/Resources/BuildStamp.txt`.** The binary-scene integration playbook commits before the regenerate, so the trap doesn't bite there ‚Äî solo asset+source work in the same session is what's exposed. **Classifier note (2026-07-02, #234 conflict fix):** under the auto-mode classifier the BLANKET `git checkout -- Assets/ ProjectSettings/` form gets DENIED (reads as a mass-discard) ‚Äî restore with an EXPLICIT per-file pathspec list of the enumerated churn files instead (same ritual, named files; `git status --porcelain` gives the list). Sub-agents following step (3) verbatim will hit the denial ‚Äî brief them with the per-file form.
- **Binary-scene PR conflicts: regenerate-on-rebase, never hand-merge.** `Boot.unity` is binary AND bootstrap-generated ‚Äî when two PRs both bake into it, the second is "not mergeable" and git cannot resolve it. ## Configurable Enter Play Mode ‚Äî domain/scene reload disabled (ticket 86ca9a39q, Erik ¬ßC / Rank 3)

**The setting (project-scoped, ships in version control).** `ProjectSettings/EditorSettings.asset` carries
`m_EnterPlayModeOptionsEnabled: 1` + `m_EnterPlayModeOptions: 3`. This is the serialized form of
**"When entering Play Mode" = "Do not reload Domain or Scene"** ‚Äî `EnterPlayModeOptions` is a `[Flags]`
enum (`None=0`, `DisableDomainReload=1<<0=1`, `DisableSceneReload=1<<1=2`; confirmed from Unity's C#
reference source `Editor/Mono/EditorSettings.bindings.cs`), so `1|2 = 3`. Because the asset lives under
`ProjectSettings/` (NOT `EditorUserSettings.asset` or per-user PlayerPrefs), it is committed and applies on
checkout across every persona's worktree ‚Äî it is a PROJECT setting, not a per-machine pref.

**Why:** domain + scene reload on every editor Play-entry costs seconds that scale with project complexity;
disabling both drops enter-play to sub-second for logic iteration. **Editor iteration only.**

**The catch ‚Äî disabling domain reload PERSISTS static state across play-entries.** Static fields and static
event handlers are NOT cleared between editor play sessions when domain reload is off, so a stale-static
singleton/handler can accumulate ‚Üí subtle editor-only bugs that don't reproduce headlessly. The fix is a
`[RuntimeInitializeOnLoadMethod(RuntimeInitializeLoadType.SubsystemRegistration)]` reset on the owning type
that clears the static back to its initial value on each play-entry.

**THE RULE (every persona, every PR): any NEW mutable runtime static field/event MUST add a
`[RuntimeInitializeOnLoadMethod(SubsystemRegistration)]` reset ‚Äî or it accumulates stale state across editor
play-entries.** This is mechanically guarded: `Assets/Tests/EditMode/StaticStateResetTests.cs`
(`EveryMutableRuntimeStatic_HasASubsystemRegistrationReset`) reflects over the whole `FarHorizon.Runtime`
asmdef and FAILS if any mutable static (excluding `const` / `static readonly` / compiler-generated /
delegate-typed) lacks a SubsystemRegistration reset on its declaring type. Add a static + forget the reset ‚Üí
that test goes red.

**The static-reset list (audited FROM SOURCE 2026-06-17, not assumed ‚Äî grep of the `FarHorizon.Runtime`
asmdef):**
- **`BuildInfo._stamp`** (lazy `string` cache of `Resources/BuildStamp.txt`) ‚Üí `BuildInfo.ResetStaticState()`,
  nulls the cache so each play-entry re-reads the stamp. **This is the ONLY mutable runtime static.**
- **`WarmthNeed`** (Erik's named candidate) ‚Äî **NO statics at all** (instance state only); no reset needed.
- **`CastawayCharacter`** (Erik's named candidate) ‚Äî only **pure static helper methods** (`ComputeFloatGap`,
  `ComputeModelGroundLocalY`, `Fmt4`); no mutable static fields; no reset needed.
- Everything else under `static` in the asmdef is a static **method**, `static class`, `const`, or
  `static readonly` immutable constant (e.g. `SurvivalHud`/`WorldLookPalette` colors, `FullscreenBoot`
  capture-flag array) ‚Üí none persist mutable state ‚Üí none need a reset.
- There are **ZERO static events** and **ZERO static singletons/collections/delegates** in the runtime asmdef.

**Caveat ‚Äî what this setting does NOT affect (verify here, don't assume it changes correctness):**
- **Headless test runs (`-runTests`) and CI builds RELOAD the domain regardless of this editor setting** ‚Äî
  they always start with fresh statics, so the EditMode/PlayMode suites + CI are UNAFFECTED. The
  `SubsystemRegistration` resets are an editor-iteration safety net, not a correctness change for CI.
- **`serve_soak.sh`'s full clean rebuild ‚Üí bootstrap ‚Üí BuildWindows chain stays the correctness path** for any
  soak/capture. Disable-reload is purely for fast in-editor logic iteration; it does NOT change shipped behavior.
- A live cross-play-entry accumulation bug is only reproducible by a human clicking Play twice in the editor;
  the EditMode guard above is a STRUCTURAL check (reset exists + clears the field), which is the testable
  surface headlessly.

## Editor-vs-runtime divergence (the #1 trap class)

- **Awake-built procedural hierarchies don't serialize.** A hierarchy assembled in `Awake()` passes editor/EditMode checks but can ship MANGLED in the player (spike iter6 "legs pointing upwards" incident). Anything that must exist in the build is built editor-time (executeMethod) and saved into the scene/prefab; runtime code only animates what's serialized.
- **EditMode tests can't see Awake-only structure** ‚Äî same root cause; give procedural builders an editor-time entry point and test THAT.
- **`AddComponent<T>()` in an EditMode test does NOT run `Awake()`** ‚Äî a field seeded only in `Awake()` keeps its serialized default in the test rig, never the runtime value. `Awake` fires on scene-load / `Instantiate`; EditMode `AddComponent` runs outside Play Mode, so that lifecycle hook never fires. Bites SCALAR fields too, not just hierarchy structure (sibling of the entry above). Occurrence 1: commit `2122c8f` (`fx.health`). Occurrence 2: PR #238 / `86cahzycp` ‚Äî `SnakeEnemy.biteBleed` (`SnakeEnemy.cs:42`, `:119-120`) left the test rig's bleed value at its serialized default, so `BiteGate_HpDelta_EqualsTierExpected_WhenTickersFrozen` saw `ActiveCount` 0 instead of the expected 1; fixed in `eab7c53` by manually wiring `MakeBleed(1.5f, 3f)` at `Assets/Tests/EditMode/SnakeAITests.cs:160` (branch `drew/86cahzycp-snake-nits`, **unmerged ‚Äî these cites are branch-only until PR #238 lands**). **Rule: never assume `AddComponent` replays `Awake()` ‚Äî any EditMode test that `AddComponent`s a MonoBehaviour with Awake-seeded fields must manually wire those fields to match `Awake`, or the test exercises stale/default state.**
- **A permanently-cached cross-component `GetComponent<T>()` result resolved in `OnEnable` can go PERMANENTLY null from test-rig add-order alone ‚Äî a FALSE-RED sibling of the false-GREEN class documented throughout this section, and the opposite-direction sibling of the AddComponent-skips-Awake entry above (there a lifecycle hook never fires; here it fires TOO EARLY relative to sibling wiring) (ticket 86cajt6jz, PR #275 pending merge, commit 4d967a6).** `HeldWeaponCycleDebug.ResolveGate()` cached `GetComponent<HeldTool>()` once and kept the result forever; because Unity runs `OnEnable` SYNCHRONOUSLY during `AddComponent`, and the `HeldBeltWeaponVisualPlayModeTests.cs:154` test-rig `SetUp` added `HeldWeaponCycleDebug` before its `HeldTool` sibling, the one-shot cache captured `null` and the renderer never enabled ‚Äî even though the sibling existed by the time anything actually used it. Two other hypotheses (a mesh-resolve ordering issue, `Time.captureDeltaTime`) were raised and REFUTED by runtime evidence before the real cause was found. Shipped scenes deserialize all sibling components together, so this is TEST-RIG-ONLY: the test suite went RED while the shipped build stayed GREEN. **Rule: never permanently cache a cross-component reference resolved in `OnEnable`/`Awake` ‚Äî re-resolve lazily WHILE it's still null/missing, rather than caching a miss.** Corollary: harden test-rig `SetUp` add-order as regression-critical ‚Äî either mirror shipped-scene simultaneity (add all siblings before any `OnEnable`/`Awake` can fire) or make production code tolerate arbitrary add-order.
- **Shipped-build capture gate exists because of this class:** editor evidence is necessary, never sufficient ‚Äî final evidence comes from the built exe.
- **Editor `Camera.Render()` in batchmode mis-renders multi-submesh URP materials** (hero-axe PR #21: head rendered flat-brown in an editor RenderTexture while the serialized materials were correct AND the shipped exe rendered barn-red). Judgment-grade prop/visual captures must come from the SHIPPED exe (serve_soak / windowed launch), never from an editor RenderTexture ‚Äî an editor capture can show a false NEGATIVE as easily as a false positive.

## UI Toolkit ‚Äî shared input-gate across independent panels

- **A single ref-counted input-gate shared by two INDEPENDENT UI Toolkit panels (e.g. an F1 settings panel + an F3 dev-console) can clear the WHOLE gate when either panel closes ‚Äî even while the OTHER panel still has a focused field ‚Äî letting typed keystrokes bleed through to world/locomotion input (observed 2026-07-03, pending PR #247 merge).** A bare shared counter (or an unconditional clear-on-close) treats "gate held" as one global boolean/count owned by whichever panel last touched it; it has no notion of WHICH panel is still open, so panel B's close path can zero the gate out from under panel A's still-focused field. **Fix: model the gate as a per-panel CLAIM-TOKEN SET, not a bare counter.** Each panel that needs the gate held adds its own identity/token to the set on open/focus-gain; the gate reads as held IFF the set is non-empty; on close a panel removes only its OWN token, so a still-open sibling panel's token keeps the gate held regardless of what any other panel does. Generalizes to any shared modal-input-gate serving more than one independent UI surface.

## UI Toolkit ‚Äî absolute-only children collapse the wrapper's auto-height (86cah8ukr, PR #247 pending merge)

A wrapper `VisualElement` whose ONLY child is `position: absolute` computes an intrinsic auto-height of ZERO ‚Äî UI Toolkit's Yoga layout excludes absolutely-positioned children from a parent's auto-size calculation, since an absolute child is removed from normal flow by design. When a settings/drawer panel nests `scrim (inset:0) ‚Üí max-height:70% container ‚Üí flex-grow ScrollView` and the scrim is the only child of an auto-height wrapper, the wrapper collapses to 0 height and the WHOLE downstream chain ‚Äî the 70%-capped container, the flex-grow ScrollView, every data row inside it ‚Äî never gets a nonzero layout box, even though the data layer underneath is completely correct. Symptom: a settings/dev-console drawer renders only its title bar + Reset button (chrome living outside the collapsed subtree) with all actual content invisible.

**Fix (PR #247, pending merge, ticket `86cah8ukr`): make the CONTAINER ITSELF the absolute-positioned overlay** ‚Äî `position: absolute; inset: 0; pickingMode: Ignore` on the outer element that used to be the auto-height wrapper ‚Äî removing the auto-height-wrapper/absolute-child split entirely, so no parent's sizing depends on an absolutely-positioned child.

**Rule: never nest a `position: absolute` element as the SOLE child of a wrapper that relies on auto/intrinsic height** ‚Äî Yoga will not size the wrapper from an absolutely-positioned child's content. If an element needs to behave as an overlay (cover its parent regardless of layout), make the OVERLAY ITSELF the absolute-positioned element with explicit bounds (`inset:0` or explicit width/height), rather than nesting it inside another auto-sized element.

## UI Toolkit ‚Äî a lone `flex-grow` child keeps `flex-shrink:1` and can collapse below its content in a no-scrollbar row (86cah8ukr, PR #247 pending merge)

A UI Toolkit `flex-grow:1` child does NOT get `flex-shrink:0` for free ‚Äî Yoga's default `flex-shrink:1` still applies, so "grows" and "won't shrink below its content" are two SEPARATE properties. In a fixed-width, no-scrollbar row where that grow-child is the ONLY shrinkable sibling (its neighbors are pinned `flex-shrink:0`), it absorbs 100% of the row's shrink pressure and Yoga squeezes it BELOW its own content's minimum width, crushing whatever it holds. Symptom (Sponsor soak of the F1 player Settings drawer, 2026-07-04): the number-stepper rows (Belt slots / Inventory stack size) rendered CRAMPED ‚Äî buttons jammed as `‚àí 5 [+ 5] 5`, Sponsor: "not enough room for the content." Root cause: `.setting-row__control` had `flex-grow:1` + the default `flex-shrink:1` and was the only shrinkable child in the row (the label + readout cells are pinned `flex-shrink:0`), so it absorbed all the shrink and collapsed below its `[‚àí]`/value/`[+]` content's ~100px minimum. Distinct mechanism from the "absolute-only children collapse the wrapper's auto-height" entry above (that's zero-height from auto-size exclusion; this is a shrink-crush) ‚Äî same drawer, same PR, two independent layout bugs.

**The identical stepper markup read FINE in the sibling F3 dev-console drawer ‚Äî a scrollbar MASKED the same bug.** F3's 62-row list triggers a scrollbar that narrows its rows past the flex-wrap breakpoint, so the def-hint text wraps to a second line and frees width back to the control before it can crush; F1's short list never grows a scrollbar, so the same underlying defect went uncompensated and was fully exposed. **"It looks fine in the other panel" is not proof a shared USS class is correct ‚Äî verify the no-scrollbar / short-list case explicitly**, since an incidental scrollbar (or any width-narrowing effect) can silently mask a flex-shrink collapse that a fixed-width sibling exposes.

**Fix (USS, stepper archetype ONLY ‚Äî PR #247 pending merge): pin the control + its cells `flex-shrink:0` + `min-width:108px`** so a too-narrow row WRAPS (like the scrollbar case did by accident) instead of crushing; sliders/ranges/toggles and the F3 drawer are untouched. Hardened by a regression gate: `verify_settings_gate.sh` Check 5 measures the smallest resolved stepper-cell width per drawer and fails under 20px (~28px is healthy), reading `settings_player_open.png` + a `PLAYER STEPPER fit: minCellWidth=‚Ä¶` log line from a new read-only `MinStepperCellWidth` probe in `SettingsPanel.cs`.

**Rule: any `flex-grow` child sharing a fixed-width row with fixed-content siblings (stepper `[‚àí]`/value/`[+]`, icon+label pairs, etc.) keeps `flex-shrink:1` by default ‚Äî pin `flex-shrink:0` + an explicit `min-width` on THAT child (not just its neighbors), or it absorbs all the row's shrink and collapses below its content.** And when a shared USS class serves more than one panel, test it in BOTH a scrollbar-present and a no-scrollbar/short-list context.

## Low-poly mesh patterns

- **Welded verts + `RecalculateNormals` = the smooth-shaded look** (averaged vertex normals over coarse facets). Import equivalents: Normals=Calculate with smoothing angle ~60¬∞, not 0¬∞.
- **Thin double-sided foliage: never reuse verts with opposite winding** ‚Äî coincident opposite-facing triangles average their normals to ~zero ‚Üí near-black "shards" (iter8 grass bug). Use distinct verts per face AND **up-bias the normals on both faces** (foliage reads as overhead-lit, not as a solid). A normal-distribution probe (count verts with `N¬∑L < threshold` against the key light) is the load-bearing verification ‚Äî the first plausible fix still shaded dark.
- **Per-instance color-jittered materials cause asset churn** ‚Äî quantize to a coarse palette and keep materials inline (not `CreateAsset`) so they serialize into the scene.
- **The coarse 12-step palette quantizer splits NEAR-NEUTRAL warm tints into a pink `R>G=B` cast** *(pending integ/soakfix7 merge ‚Äî `QuantizeFine`, drew/rocks-sourced @ `d358176`, ticket 86ca8m5zu).* The coarse quantizer (above) snaps each channel to 12 steps ‚Äî fine for saturated colours, but a near-grey warm ramp (R‚âàG‚âàB, low chroma) snaps to `(0.667, 0.583, 0.583)` = R>G=B = unmistakable PINK. Diagnostic: a `TINTDIAG` material dump (the mesh/UV are fine; the delivered `_Color` is wrong). Fix: a finer 24-step `QuantizeFine` for low-chroma / near-grey props (preserves R‚â•G‚â•B; ~4 mats vs the 1 that pinked). Rule of thumb: if a prop's intended tint has all channels within ~0.1 (chroma < ~0.1), use the fine quantizer; saturated props stay safe on the coarse step.
- **A ‚àíZ procedural grid that reuses a +Z grid's triangle winding ships INVERTED normals ‚Üí backface-culled / invisible (Drew PR #38, 2026-06-13).** The sea grid lays its rows near‚Üífar in DECREASING world Z (the sea runs AWAY from the +Z shore) but reused the terrain grid's triangle index order (written for INCREASING Z). Same index order + mirrored vertex layout = faces wound the opposite way ‚Üí ‚àíY normals ‚Üí the default URP `Cull Back` culls them from any above-looking camera. The water was NEVER occluded ‚Äî it was backface-culled (proven: a `-seaWaterOnly` probe that hides every other mesh STILL rendered 0 sea px; the magenta-diff showed 0‚Üí55,103 px / 5.98% after the fix). Fix = reverse the triangle winding for the ‚àíZ grid (`LowPolyZoneGen.BuildWaterEdge`). Guard = assert every face normal points the expected way (`WaterFacesUpTests`: every `Water_Play` normal¬∑+Y > 0) ‚Äî a normal-direction guard that runs in EditMode with NO build, the silhouette/normal class. Same family as the foliage opposite-winding bug above ‚Äî whenever you reuse a grid's index buffer for a grid laid out in a different direction, re-derive or flip the winding.
- **ANY procedural FLAT-SHADED mesh with explicit per-face normals must enforce OUTWARD winding or URP `Cull Back` silently culls it ‚Äî same class as the ‚àíZ grid bug above** *(pending integ/soakfix7 merge ‚Äî `FacetedRock`, drew/rocks-sourced @ `d358176`, ticket 86ca8m5zu).* Flat shading needs unwelded verts so each face carries its own baked normal; faces wound INWARD ‚Üí ‚àínormals ‚Üí culled from every camera ‚Üí the mesh renders as a thin sliver (or nothing) despite being geometrically chunky. **Diagnostic to tell "culled" from "actually-thin":** a geometry-dump probe (`ROCKDIAG`) reporting per-face extent ratios ‚Äî ratio ~0.9 + 0 thin faces means the mesh IS chunky, so invisibility is culling, not geometry. Fix: force every generated face normal outward + flip winding; guard `FacetedRock_AllFacesPointOutward` (EditMode, no build). Pattern: when a builder bakes per-face normals explicitly (instead of `RecalculateNormals`, which self-corrects outward), verify winding independently ‚Äî and note the EDITOR may still draw the culled mesh, masking the runtime cull until the shipped build.
- **A grid-cell CUTOUT (or its guard) needs a triangle-OVERLAP predicate, NOT a per-cell CENTROID test, when cells are coarser than the cut footprint (pond #130, 2026-06-25).** Cutting a feature footprint (pond outline, hole, mask) out of a procedural grid by testing each cell's CENTROID against the footprint leaves SLIVERS: a cell whose centre falls outside the outline but whose body still overlaps it is kept, so a partial cell juts into the hole. Worse, if the cut uses an overlap test but the GUARD that asserts "the hole is clean" uses a centroid test (or vice-versa), the guard SILENT-PASSES the slivers ‚Äî the predicate mismatch hides the defect. Rule: the cut and its verification must use the SAME predicate, and for any grid whose cell size can EXCEED the cut footprint that predicate must be triangle/cell-OVERLAP (does the cell's extent intersect the footprint at all), not centroid-inside. Same predicate-mismatch / proxy-guard family as the cull-by-winding-vs-normal entries above.
- **The castaway model is Y-yaw-only ‚Äî no body tilt/lean/roll exists anywhere in the rig, so a "lean into the turn" or "tilt on landing" ask is NEW work, not a tuning knob.** `CastawayCharacter.cs:1182-1185` drives facing purely as `_bodyYaw = Mathf.LerpAngle(_bodyYaw, target, 1f - Mathf.Exp(-turnLerp * Time.deltaTime))` then `_model.localRotation = Quaternion.Euler(0f, _bodyYaw, 0f)` ‚Äî pitch and roll are hardcoded `0f`, never touched anywhere else in the class. `CastawayArmPose` (the project's one arm-modification idiom, see `procedural-animation-verbs.md`) only offsets arm/hand bones ‚Äî it never reaches the torso. **Consequence:** never scope a visible lean/tilt ticket as "adjust the existing tilt" or "extend CastawayArmPose" ‚Äî there is nothing to extend. It requires a NEW torso/spine-roll driver sibling to `CastawayArmPose` (its own additive `LateUpdate` offset on `_model` or a spine bone), which is itself a new animation idiom, not a parameter change.
- **The air-control speed cap does not brake a run-jump ‚Äî it only bites a standing jump ‚Äî because the cap floor is `max(airControlMaxSpeed, startSpeed)`, not a flat ceiling.** `WasdMovement.AirborneVelocity` (`WasdMovement.cs:356-382`, called from `:254-255`) does `MoveTowards(horizVel, dir * Mathf.Max(airControlMaxSpeed, startSpeed), airControlAccel * dt)` ‚Äî `airControlMaxSpeed` is the walk-speed default (5.5, `:77`), but a run-jump's carried-in `startSpeed` (~9.5) exceeds it, so the effective cap RISES to the carried-in speed and steering never decelerates a running jump. Only a jump launched from a near-standstill sees the 5.5 cap actually constrain it. (`airControlAccel`, the per-frame steer-rate tunable, is mid-tuning on an unmerged PR as of 2026-07-01 ‚Äî don't cite its current value in tickets or docs; re-check `WasdMovement.cs` for the live number.) **Consequence:** a "jump feels too floaty/steerable" bug report needs to specify WHICH jump (standing vs run-jump) before diagnosing ‚Äî the same code path produces very different felt air-control depending on entry speed, and "lower `airControlMaxSpeed`" is a no-op fix for the run-jump case.
- **Landing snaps facing to the currently-held direction at full speed with zero transition smoothing ‚Äî this is the shipped design, not a bug to smooth out.** The first grounded frame after landing runs `_agent.velocity = LastMoveDir * speed` (`WasdMovement.cs:259`, `LastMoveDir` latched from held input at `:226`); `CastawayCharacter` reads that velocity for facing and eases `_bodyYaw` toward it via the `LerpAngle` above (`:1182-1185`) ‚Äî so the BODY turn is damped but the underlying MOVE direction itself snaps instantly on touchdown, with no separate landing-transition state or blend anywhere in either file. **Consequence (compounds with the two facts above):** a "landing feels abrupt / turns funny" ticket is not fixable by tuning one system in isolation ‚Äî the felt result is the composition of (a) the instant velocity snap on the grounded frame, (b) the yaw-only lerp giving the ONLY smoothing that exists, and (c) the air-control cap doing nothing during a run-jump's descent. A full landing-feel pass needs to consider tilt/lean (fact 1), air-control (fact 2), and this snap (fact 3) together ‚Äî fixing just one in isolation will not change the composite feel the Sponsor is judging.

## CI architecture ‚Äî merge-gate vs advisory PlayMode (PR #80 `86caapwmt`, 2026-06-18)

- **~~The self-hosted runner has a CONFIRMED play-mode-ENTER env-deadlock~~ ‚Äî REFUTED 2026-07-03 (PR #249 empirical bisection, pending merge): the "environmental / fires-before-any-fixture" conclusion was WRONG. The real root cause was a CODE bug in the tests:** 10 locomotion/scene fixtures shared an isolation `[UnitySetUp]` that did `UnloadSceneAsync(s); while(!op.isDone) yield return null` over EVERY loaded scene ‚Äî INCLUDING the Unity test framework's own `InitTestScene`, whose async unload never completes headless ‚Üí an unbounded wait spun forever at the FIRST such fixture (Airborne), masquerading as a play-mode-ENTER deadlock (the log freezes at "Entering Playmode" because the hang is in the first fixture's setup, before any test body logs). The earlier hypotheses ‚Äî environmental HID/USB, UUM-142421 (persisted after the 6000.4.11f1 upgrade), Android module ‚Äî were ALL refuted by A/B bisection on the actual runner ([[verify-hang-rootcause-against-actual-machine]] vindicated). Fix (PR #249): a shared `PlayModeSceneIsolation` helper that SKIPS `InitTestScene` + BOUNDS the unload wait (fails loud on timeout), refactored into the 10 fixtures ‚Äî the full PlayMode suite then completes in ~200s. **Lessons:** (1) `UnloadSceneAsync` over ALL loaded scenes in a test-setup must skip the framework's `InitTestScene` and must bound its wait; (2) a hang that "fires before any fixture runs" may actually be IN the first fixture's `[UnitySetUp]` ‚Äî the enter-phase log boundary lies; (3) "environmental, not a code bug" is a hypothesis until a bisection on the real machine proves it.
- **Why a step-level `timeout-minutes` + `continue-on-error` on an in-line PlayMode step does NOT keep the job green (proven on run `27769261766`):** the hung headless Unity IGNORES the GitHub step-cancel that `timeout-minutes` fires, so the orphaned `Unity.exe` keeps the runner pinned until the runner **LOSES COMMUNICATION with the server** ‚Äî an INFRASTRUCTURE failure (`The self-hosted runner lost communication with the server`) that `continue-on-error` CANNOT suppress (it catches step exit codes, not a lost runner). The in-line PlayMode step killed the WHOLE `unity` job ‚Äî Build + capture never ran ‚Äî turning the merge gate red on an env problem.
- **The structural fix ‚Äî split the workflow into a reliable blocking gate + an advisory PlayMode job:**
  - **Merge gate = `structure` (hosted) + `unity` (self-hosted): bootstrap ‚Üí EditMode ‚Üí BuildWindows ‚Üí shipped-build capture.** All three Unity gates are reliable; all BLOCKING. The `unity` job no longer runs PlayMode at all.
  - **`playmode` = a SEPARATE, ADVISORY, NON-REQUIRED, FAIL-FAST job.** `continue-on-error: true` at the JOB level ‚Üí it can NEVER block a merge (do NOT add its check to required status checks). `timeout-minutes: 20` is a HARD cap so a FUTURE play-mode-enter deadlock fails FAST + NAMED (the `playmode` job goes red, clearly attributable) instead of hanging to the 90-min job cap and crashing the runner. `needs: unity` SERIALIZES it AFTER the merge-gate build so the two Unity invocations never contend the single runner's PackageCache (EPERM ‚Äî `[[single-unity-build-slot-serializes-orchestration]]`). `if: always()` lets it run for signal even when the merge-gate job failed.

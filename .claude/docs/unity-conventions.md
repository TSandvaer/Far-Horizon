# Unity Conventions — hard-won findings (eval spike + bootstrap harvest)

Findings proven empirically during the Embergrave engine-eval spike (`c:/Trunk/PRIVATE/EmbergraveUnitySlice`, iter1–8, RandomGame tickets `86ca7y46c` / `86ca7zhyk` / `86ca7zkyr`) and the Far Horizon bootstrap (U1 `86ca86fb7`, root commit `3a6ef5c`). Each cost at least one debugging round to learn — read before touching the relevant surface.

## Headless / CLI rituals

- **Runtime captures must run WINDOWED, not `-batchmode`.** `-batchmode` produces no real rendered frames; screenshot evidence comes from launching the built exe windowed (`-screen-fullscreen 0`) with an in-game capture component.
- **Editor automation pattern:** `Unity.exe -batchmode -quit -projectPath <p> -executeMethod <Class.Method> -logFile <log>` for scene/asset work; `-runTests -testPlatform EditMode|PlayMode -testResults <xml>` for tests (NO `-quit` with `-runTests`). Always grep the XML's `<test-run ... result= total= passed= failed=` line — exit code alone lies on some failure classes.
- **Headless PlayMode time trap:** `Time.deltaTime ≈ 0` per frame in headless runs — never assert on per-frame deltas; sample over a real `Time.time` window instead.
- **Local `-batchmode -runTests -testPlatform PlayMode` can DEADLOCK at "Entering Playmode" on this machine — EditMode is unaffected; CI is the authoritative PlayMode gate (observed by Devon during the `86caac81y` air-control work, 2026-06-17).** Reproducible on a CLEAN tree before any test runs → it's machine/environment state, NOT the code. Two gotchas: (1) a `taskkill` of a hung PlayMode run leaves stray `Assets/InitTestScene*.unity` files that RE-HANG the next run — delete them before retrying. (2) `serve_soak.sh` does NOT run PlayMode, so soak builds are unaffected. When a local PlayMode run deadlocks, don't fight it — run EditMode locally and let the PR's CI run be the authoritative PlayMode result (the self-hosted runner runs the identical command on a fresh checkout and passes).
- **`WaitForEndOfFrame` is NOT evoked in `-batchmode` (Devon, PR #60 locomotion-harness work).** A PlayMode coroutine `yield return new WaitForEndOfFrame()` never resumes headlessly (there is no end-of-frame render pass to fire it), so any value you intended to sample AFTER `LateUpdate`/render is never read — the test hangs or silently skips the assertion. To sample a post-`LateUpdate` value in a headless PlayMode coroutine, record it in a high-`DefaultExecutionOrder` `LateUpdate` component and read the PRIOR frame's recorded value on the next `yield return null` — never gate the sample on `WaitForEndOfFrame`.
- **Build verification:** the builder must exit non-zero on failure and log `result=Succeeded size=<bytes>`; grep that line, never trust silence.
- **Build-stamp ritual:** HUD shows `BUILD <tag> | <UTC> | <sha>`; every soak/judgment verifies the stamp first (three-builds-in-play identity confusion is a proven failure mode).
- **Stale-stamp trap (the soak handoff):** `FarHorizonBuilder.BuildWindows` does NOT regenerate `Assets/Resources/BuildStamp.txt` — ONLY `BootstrapProject.Run` (`WriteBuildStamp`) writes it, and the stamp is COMMITTED, so it is ALWAYS stale vs a later HEAD. A soak that runs a bare `BuildWindows` ships the old committed sha → build-identity ambiguity (bit the first manual desktop soak, 2026-06-12). **Never serve a soak with a bare build.** Use the one entry point `.github/workflows/scripts/serve_soak.sh` (ticket 86ca86gde): from a CLEAN checkout it chains bootstrap (fresh-stamps HEAD) → BuildWindows → `verify_build_stamp.py` (fails loud unless the shipped stamp's sha == HEAD) → `capture_gate.sh` (reuses U7's frame gate) → restores the bootstrap's tracked-asset churn (`git checkout -- Assets ProjectSettings`; the clean-tree precondition makes whole-root restore complete + drift-proof — bootstrap's churn set drifts as it grows, so an enumerated path list goes stale). Prints the exe path + HUD stamp + capture dir handoff block. Regression-guarded by `verify_build_stamp.py` + the stale-vs-HEAD case in `tests/scripts/test_gate_scripts.sh`.
- **Capture-evidence freshness: cited == attached == HEAD.** Recurring failure class (3× on 2026-06-12: PR #10 stale-stamp PNGs, PR #13 capture-not-attached): a Self-Test Report CITES a capture/stamp, but the artifact on disk is from an older sha, or lives only in gitignored `ci-out/`/`Captures/` where the reviewer can't see it. The gate rests on report text alone — exactly what the capture gate exists to prevent. Convention: (1) the capture's HUD stamp must read the PR HEAD sha at citation time; (2) include the verify-run's exit-code log line verbatim in the PR; (3) image attachment to PRs is structurally IMPOSSIBLE from agents (gitignore + structure_check.sh gate committed captures; gist upload classifier-denied for private-repo screenshots; gh has no comment-attachment API) — the canonical verification is the REVIEWER INDEPENDENTLY RE-RUNNING the verify gate (serve_soak.sh / -verifyChop class) and judging their own artifact (PR #13 precedent, 2026-06-12). Reviewers: always check cited-stamp == artifact-stamp == HEAD before trusting any visual claim. Same class, CI flavor (4th hit, PR #14): "CI green" claims must cite the actual run ID on the PR head — local EditMode/PlayMode/serve_soak results are NOT CI; a PR body asserting CI-green while the head's only run is red fails the evidence gate even when the code is approve-quality. Self-hosted-runner note (CONFIRMED 2× 2026-06-12, runs 27433014608 + PR #15 first attempt): concurrent Unity instances collide on PackageCache (`EPERM rename` in bootstrap) → environmental CI red. Policy: SERIALIZE CI-triggering pushes (don't push two PRs back-to-back while the runner is busy); on a hit, one re-run — it clears when the lock holder exits. The SAME collision hits a dispatched QA REVIEWER's independent-reproduction build: their `serve_soak` runs Unity in their worktree on the same self-hosted-runner machine, so it races an active CI build → EPERM for one of them. When you dispatch a reviewer who will build while the PR's CI is still running, message them to HOLD the shipped-build step until CI clears (do code review meanwhile), then release them (near-miss 2026-06-13, PR #26: Tess dispatched into a live unity CI run; serialized by message before her build started). The `gh run list` (no-in-flight-CI) check is necessary but NOT sufficient (PR #36): a still-RESIDENT LOCAL Unity/exe process — from a just-finished `serve_soak` or local test that hasn't fully exited — can hold the PackageCache lock and trigger the same EPERM even when no CI run is in-flight. Let local Unity/exe processes fully EXIT before pushing a CI-triggering commit. Unity incremental builds can leave the old exe mtime unchanged even on a successful rebuild. Always grep the build log for `result=Succeeded` (plus `size=<bytes>`); use the `BurstDebugInformation_DoNotShip/` folder mtime as a secondary freshness check when the log is unavailable.
- **A shipped-exe capture can STILL be FALSE-GREEN if its camera/lighting ≠ gameplay (86ca8ca1m soak, 2026-06-13).** The castaway recolor passed Tess's isolated `-verifyCastaway` shipped capture (a purpose-built hero framing) AND every guard — yet the Sponsor's GAMEPLAY orbit-cam view washed the character to a yellow blob (the scene's warm key + bloom blew the bright atlas out; the isolated verify rig didn't reproduce that lighting). Even a real shipped-exe frame lies if the `-verify<X>` mode frames/lights the subject differently than actual play. Lesson: judgment-grade VISUAL captures must use the GAMEPLAY camera (the orbit cam at its real pitch) + the actual scene lighting/post — not a flattering isolated hero shot. Reviewers + capture gates should prefer a gameplay-orbit frame; an isolated verify capture is a smoke-test that the asset EXISTS, NOT proof of how it READS in play. (This is the sibling of the editor-vs-shipped false-negative below: same trap, capture-rig flavor.) **To BUILD a trustworthy verify capture (PR #37 recipe):** ride the gameplay render path (post-processing ON + skybox/scene ambient); frame from settled `SkinnedMeshRenderer.BakeMesh` bounds — NEVER raw `.bounds` or a `Mathf.Max` height floor (invalid skinned bounds → non-deterministic distance → cropped/rear shots); pin a KNOWN facing (don't assume +Z=front); use a NEUTRAL clear backdrop (a warm/bright clear blows the subject's R-channel to cream); and UNIT-TEST the framing math (the bug slipped 3 PRs precisely because the framing math was untested). Built this way the capture is deterministic (byte-identical across runs) AND gameplay-representative.
- **"Can't see X in the build" is often a CAMERA-reach / occlusion bug, NOT an asset bug (PR #35 ocean soak-fix, 2026-06-13).** The Sponsor's "I see no ocean" + the ticket both blamed the water material — but the water shipped CORRECT. Real causes: (a) the orbit-cam pitch FLOOR (minPitch 35) never tilted low enough to frame the sea, and (b) a beach dune crest occluded the water between the locked spawn and the waterline. Before re-authoring an asset that "doesn't render," diagnose-via-trace: where does the gameplay camera actually point (pitch/zoom clamp range)? is any geometry between the camera and the target? Fixing the camera reach + removing the occluder beat re-tuning the material. Pairs with the false-green-capture trap above — both are "the asset is fine, the VIEW is the problem."
- **`BootstrapProject.Run` dirties tracked assets.** Running bootstrap dirtied Boot.unity, several materials, GraphicsSettings.asset, and BuildStamp.txt in observed runs. Always follow bootstrap with `git checkout -- Assets/ ProjectSettings/` before committing or opening PRs — bootstrap churn must not ship in unrelated work.
- **`git checkout -- Assets/ ProjectSettings/` ALSO reverts UNCOMMITTED source `.cs` edits under `Assets/Scripts/` (Devon, 2026-06-14).** That restore is a blanket discard of every tracked file under `Assets/` — including your in-flight `.cs` source — so running bootstrap + restore mid-task silently wipes any source edits you haven't committed (Devon lost all 5 mid-task edits this way; caught only because a verify-grep returned 0). Safe pattern for any worktree that mixes source edits with a bootstrap regenerate: **(1) COMMIT the `.cs` work first, (2) run `BootstrapProject.Run` headless, (3) restore with `git checkout -- Assets/ ProjectSettings/` + surgically re-add only `Assets/Scenes/Boot.unity Assets/Resources/BuildStamp.txt`.** The binary-scene integration playbook commits before the regenerate, so the trap doesn't bite there — solo asset+source work in the same session is what's exposed.
- **Binary-scene PR conflicts: regenerate-on-rebase, never hand-merge.** `Boot.unity` is binary AND bootstrap-generated — when two PRs both bake into it, the second is "not mergeable" and git cannot resolve it. Validated fix (PR #10 after PR #11 moved main, 2026-06-12): (1) `git rebase origin/main`, resolving CODE conflicts normally but taking `--theirs` provisionally on Boot.unity/BuildStamp.txt; (2) re-run `BootstrapProject.Run` headless — the regenerated scene bakes BOTH branches' contributions because both live in the bootstrap code path; (3) prove the carry with EVERY feature's scene-presence EditMode test green (e.g. `WarmthNeedSceneTests` + `CastawayCharacterTests` together); (4) fresh `serve_soak.sh` so captures/stamp match the new HEAD; (5) `git push --force-with-lease`, keeping the committed set surgical (Boot.unity + BuildStamp.txt only). Reviewer content-APPROVEs survive this — it's mechanical integration. Sequencing corollary: every scene-touching PR conflicts with every other; merge them tightly rather than letting approved PRs queue. Batch corollary (PR #32, 2026-06-13): when 2+ scene PRs conflict at once, integrate them in ONE branch (merge all their code + regenerate Boot.unity once) rather than N separate rebases — cheaper, and the single shipped-build verify doubles as the combined soak. STALE-BRANCH DEAD-CODE trap: a branch that forked BEFORE another PR merged can re-introduce code that PR deleted (PR #31's diff context re-added `MakeAxeMat` that PR #29's sourced-axe re-do had already deleted from main) — when resolving the code merge, drop anything now orphaned (zero callers) instead of re-committing dead code.

## Configurable Enter Play Mode — domain/scene reload disabled (ticket 86ca9a39q, Erik §C / Rank 3)

**The setting (project-scoped, ships in version control).** `ProjectSettings/EditorSettings.asset` carries
`m_EnterPlayModeOptionsEnabled: 1` + `m_EnterPlayModeOptions: 3`. This is the serialized form of
**"When entering Play Mode" = "Do not reload Domain or Scene"** — `EnterPlayModeOptions` is a `[Flags]`
enum (`None=0`, `DisableDomainReload=1<<0=1`, `DisableSceneReload=1<<1=2`; confirmed from Unity's C#
reference source `Editor/Mono/EditorSettings.bindings.cs`), so `1|2 = 3`. Because the asset lives under
`ProjectSettings/` (NOT `EditorUserSettings.asset` or per-user PlayerPrefs), it is committed and applies on
checkout across every persona's worktree — it is a PROJECT setting, not a per-machine pref.

**Why:** domain + scene reload on every editor Play-entry costs seconds that scale with project complexity;
disabling both drops enter-play to sub-second for logic iteration. **Editor iteration only.**

**The catch — disabling domain reload PERSISTS static state across play-entries.** Static fields and static
event handlers are NOT cleared between editor play sessions when domain reload is off, so a stale-static
singleton/handler can accumulate → subtle editor-only bugs that don't reproduce headlessly. The fix is a
`[RuntimeInitializeOnLoadMethod(RuntimeInitializeLoadType.SubsystemRegistration)]` reset on the owning type
that clears the static back to its initial value on each play-entry.

**THE RULE (every persona, every PR): any NEW mutable runtime static field/event MUST add a
`[RuntimeInitializeOnLoadMethod(SubsystemRegistration)]` reset — or it accumulates stale state across editor
play-entries.** This is mechanically guarded: `Assets/Tests/EditMode/StaticStateResetTests.cs`
(`EveryMutableRuntimeStatic_HasASubsystemRegistrationReset`) reflects over the whole `FarHorizon.Runtime`
asmdef and FAILS if any mutable static (excluding `const` / `static readonly` / compiler-generated /
delegate-typed) lacks a SubsystemRegistration reset on its declaring type. Add a static + forget the reset →
that test goes red.

**The static-reset list (audited FROM SOURCE 2026-06-17, not assumed — grep of the `FarHorizon.Runtime`
asmdef):**
- **`BuildInfo._stamp`** (lazy `string` cache of `Resources/BuildStamp.txt`) → `BuildInfo.ResetStaticState()`,
  nulls the cache so each play-entry re-reads the stamp. **This is the ONLY mutable runtime static.**
- **`WarmthNeed`** (Erik's named candidate) — **NO statics at all** (instance state only); no reset needed.
- **`CastawayCharacter`** (Erik's named candidate) — only **pure static helper methods** (`ComputeFloatGap`,
  `ComputeModelGroundLocalY`, `Fmt4`); no mutable static fields; no reset needed.
- Everything else under `static` in the asmdef is a static **method**, `static class`, `const`, or
  `static readonly` immutable constant (e.g. `SurvivalHud`/`WorldLookPalette` colors, `FullscreenBoot`
  capture-flag array) → none persist mutable state → none need a reset.
- There are **ZERO static events** and **ZERO static singletons/collections/delegates** in the runtime asmdef.

**Caveat — what this setting does NOT affect (verify here, don't assume it changes correctness):**
- **Headless test runs (`-runTests`) and CI builds RELOAD the domain regardless of this editor setting** —
  they always start with fresh statics, so the EditMode/PlayMode suites + CI are UNAFFECTED. The
  `SubsystemRegistration` resets are an editor-iteration safety net, not a correctness change for CI.
- **`serve_soak.sh`'s full clean rebuild → bootstrap → BuildWindows chain stays the correctness path** for any
  soak/capture. Disable-reload is purely for fast in-editor logic iteration; it does NOT change shipped behavior.
- A live cross-play-entry accumulation bug is only reproducible by a human clicking Play twice in the editor;
  the EditMode guard above is a STRUCTURAL check (reset exists + clears the field), which is the testable
  surface headlessly.

## Asset creation — Blender + Blender MCP (Sponsor-flagged capability, 2026-06-12)

The Sponsor runs Blender locally with the **Blender MCP** connected to orchestrator sessions
(`mcp__blender__*` tools; sub-agents load them via ToolSearch). This is the first-choice route for
CREATING low-poly objects, not just importing CC0 packs:

- **Procedural low-poly modeling:** `execute_blender_code` scripts meshes directly — ideal for the
  faceted/flat-shaded style (props, rocks, trees, campfire variants); `get_viewport_screenshot` for
  iteration; export FBX via scripted exporter into `Assets/Art/`.
- **Asset sources:** PolyHaven (CC0 download tools) + Sketchfab search/import; Hyper3D/Hunyuan3D
  text- or image-to-3D generation for rough bases to retopo/stylize.
- **Fit to pipeline:** exported FBX goes through the SAME editor-time import/serialization rituals
  as the Quaternius castaway (importer config + bake into scene/prefab; all trap classes apply).
- **Named use cases so far:** future world props (the campfire visual bar, Zone-D trees/landmarks
  per the art board); rough generated bases to retopo.
- **Connecting the MCP:** the addon must be enabled INSIDE Blender (Edit → Preferences → Add-ons →
  enable `blender-mcp-addon`); enabling it auto-starts the server thread (no separate N-panel
  "Connect" click is strictly required — `addon_enable` logs `Server thread started`). The
  orchestrator cannot click in Blender — the Sponsor enables it; then `get_scene_info` verifies.
  Workbench renders only show material colors with `scene.display.shading.color_type='MATERIAL'`.
- **PROPORTION EDITS ON A RIGGED CHARACTER → BONE-SCALE, NOT VERTEX SURGERY (86ca8ca1m, hard-won
  2026-06-13).** Blind `execute_blender_code` vertex deformation of a rigged humanoid (radial
  fatten, collapse-fingers-to-a-point, vertical squash) MANGLES limbs/hands — fingers crumple,
  forearms become thin blades, eyeballs balloon — and the Sponsor rejected it outright ("the arms
  and hands are destroyed"). The clean route for chunkier/shorter proportions is **per-bone
  baseline scale in Unity** (uniform scale on a bone; the skinned mesh follows the weights, so
  limbs never tear and Idle/Walk survive). Reserve Blender mesh edits for things bone-scale CAN'T
  do (material-slot splits, added geometry), and even then move FACES/materials, not bulk verts.
  Note this REVISES Uma's `castaway-style-v2.md` §5 "edit the existing rigged mesh" recommendation
  — bone-scale won empirically.
- **`remove_doubles` / merge-by-distance welds the WHOLE mesh, not just your selection** if the
  threshold is large — a 0.22 threshold on a ~5u-tall mesh collapsed 3980→391 verts and destroyed
  it. There is no cheap undo over MCP; re-import the source FBX and replay the good edits. Keep
  thresholds tiny (≤0.01) and verify the selection count first, or avoid the op.
- **SOURCE a purpose-built base when the existing mesh fights the target style (86ca8ca1m).** A whole
  evening of trying to cartoon-ify the realistic Quaternius head failed (face won't take big
  white-sclera/black-pupil eyes + bushy brows on a socket-set realistic skull). The win was to
  Sketchfab-source a pre-rigged chunky-cartoon base ("Mini Chibi Kid", joaobaltieri, CC-BY) that
  ALREADY had the look. Heuristic: if 2+ mesh-edit attempts at a style change fail, stop editing and
  source a base built in that style. Vet candidates for: license (prefer CC-BY; AVOID
  CC-NonCommercial for a potentially-commercial game), low face count, AND — critically — whether it
  **ships its own animation set**. The chibi bundled Idle/Walk/Run+more on a Mixamo-style humanoid
  rig, which sidesteps the retarget problem (our Quaternius clips are Generic `HumanArmature|`-prefixed
  and don't transfer to a different skeleton).
- **Sketchfab/Hyper3D/PolyHaven enable is a SCENE prop, settable headlessly** — no manual N-panel
  click. `bpy.context.scene.blendermcp_use_sketchfab=True` + `...blendermcp_sketchfab_api_key="<key>"`
  (siblings: `blendermcp_use_hyper3d`/`blendermcp_hyper3d_api_key`, `blendermcp_use_polyhaven`,
  `blendermcp_use_hunyuan3d`/`..._secret_id`/`..._secret_key`). After setting, `get_sketchfab_status`
  reports enabled. The orchestrator can't click in Blender but CAN set these via `execute_blender_code`
  once the Sponsor provides the key. **The AI GENERATORS still need a key even when "enabled":**
  Hyper3D Rodin's `MAIN_SITE` mode is NOT a keyless free trial in this addon build — after
  `use_hyper3d=True` the status still reports "enabled, but API key is not given" until
  `blendermcp_hyper3d_api_key` is set; Hunyuan3D needs a Tencent `secret_id`+`secret_key` pair.
  Sketchfab (sourcing existing assets) is the only one that works with just an account API key;
  generation (Rodin/Hunyuan) is gated on the Sponsor supplying a generator key. **Hyper3D's API key
  itself requires their $96/mo BUSINESS tier** — the $30/mo Creator tier (what the Sponsor has) does
  NOT grant API access ("Unlock with Business Subscription" on the create-key button), so the Blender
  MCP cannot drive Hyper3D on a Creator plan. Creator DOES include web-app generation + "Unlimited
  Export", so the project's chosen route (2026-06-13) is: **Sponsor generates on hyper3d.ai's web UI +
  exports the FBX, orchestrator integrates it like any sourced asset** — no Business upgrade, no MCP
  API. Default remains Sketchfab sourcing (free, CC-BY); paid API-gen is only worth it for bespoke
  shapes Sketchfab can't supply.
- **Sketchfab search ANDs the query terms** — multi-word queries (`stylized hero big head rigged toon`)
  return 0–2 hits; use 1–2 keyword queries (`chibi character rigged`, `toon character rigged`) for
  large result sets, then filter visually with `get_sketchfab_model_preview` (returns a thumbnail).
  Pass `downloadable=True`.
- **The Sketchfab download temp dir is cleaned after import** — the extracted FBX + textures live in a
  `tmp_*` dir that's removed once imported; the texture ends up packed in Blender memory and the source
  FBX is NOT retrievable from disk. To preserve a downloaded asset for the repo, save the packed image
  (`img.filepath_raw=...; img.save()`) and `export_scene.fbx` from Blender — don't expect to copy the
  original download. Trim duplicate `.001-.004` import-take actions before export (`bake_anim_use_all_actions=True`).
- **Stray textures leak between asset exports in a multi-import session** — `bpy.data.images` accumulates
  every imported model's textures, and `export_scene.fbx(path_mode='COPY')` (plus a blanket
  "save all packed images" loop) copies ALL of them into the new asset's folder, not just the current
  model's. Hit on the axe export: it dropped the prior chibi's `mini_material_baseColor.png` into
  `Assets/Art/Props/CastawayAxe/`. Either clear unused images / start a fresh scene between asset
  sources, or scope the texture-save to the current model's materials, then `rm` strays from the
  output folder before committing.
- **Recoloring a sourced gradient-palette toon-atlas to project identity → repaint UV CELLS, not a material tint (86ca8ca1m / Devon PR #31).** The chibi's colors live in a 256² baseColor atlas where each mesh region maps to a UV cell carrying a baked light→dark toon gradient. A per-material color/tint FLATTENS that gradient to one flat color (kills the toon read). The right move: a **luma-PRESERVING hue remap** of the target cells (shift hue/sat, keep each texel's luma) — preserves the gradient. Derive the region→cell map empirically first (a bone-influence UV probe), then repaint (e.g. PIL). Pin the result with an identity guard (a shirt-luma anchor / cell-resample assert) so a re-import can't silently revert it — and re-add identity guards a SOURCED base lacks (the chibi shipped without the shirt-luma >0.6 guard the Quaternius castaway had). **The repaint MUST be reproducible-from-code** — an idempotent editor `Recolor*` method run at bootstrap, NOT a one-off hand-run PIL edit on the committed PNG. The v1 castaway recolor was an uncommitted hand-edit with no regeneration path: it would silently vanish on any atlas re-import and couldn't be re-tuned or re-verified. The bootstrap-run recolor regenerates the atlas deterministically every build (86ca8ca1m / PR #34).

## Editor-vs-runtime divergence (the #1 trap class)

- **Awake-built procedural hierarchies don't serialize.** A hierarchy assembled in `Awake()` passes editor/EditMode checks but can ship MANGLED in the player (spike iter6 "legs pointing upwards" incident). Anything that must exist in the build is built editor-time (executeMethod) and saved into the scene/prefab; runtime code only animates what's serialized.
- **EditMode tests can't see Awake-only structure** — same root cause; give procedural builders an editor-time entry point and test THAT.
- **Shipped-build capture gate exists because of this class:** editor evidence is necessary, never sufficient — final evidence comes from the built exe.
- **Editor `Camera.Render()` in batchmode mis-renders multi-submesh URP materials** (hero-axe PR #21: head rendered flat-brown in an editor RenderTexture while the serialized materials were correct AND the shipped exe rendered barn-red). Judgment-grade prop/visual captures must come from the SHIPPED exe (serve_soak / windowed launch), never from an editor RenderTexture — an editor capture can show a false NEGATIVE as easily as a false positive.
- **`VolumeProfile.Add<T>()` components aren't serialized into the profile asset.** Post-processing overrides added via editor code (`profile.Add<Bloom>()` etc.) live only in memory — without `AssetDatabase.AddObjectToAsset(component, profile)` per component plus a save, the profile ships EMPTY and the whole post stack silently goes missing in the build (symptom: bloom/grading/vignette present in the editor, absent in the exe only). Regression-guarded by asserting the saved profile's component count (U5, Far Horizon PR #4).
- **`SkinnedMeshRenderer.bounds` is STALE on a deserialized, never-rendered scene** — it returns the baked-at-import bounds and ignores a runtime/bake-time bone-baseline scale, so a proportion guard reading `.bounds` passes in a fresh in-session build but measures the UN-scaled ratio on the shipped (deserialized) scene. Use `SkinnedMeshRenderer.BakeMesh(mesh, useScale:true)` and measure the baked mesh for any proportion/height/bounds assertion that must survive scene deserialization (86ca8ca1m / Devon PR #25 `MeasureHeadsTall`).
- **Component-in-source-but-not-serialized-into-scene** is a named failure class alongside Awake-no-serialize: a MonoBehaviour can be committed, compile clean, and pass all script tests while the scene simply never carries it — the feature ships silently inert (U7, PR #6 `a3edf04`: `CaptureGate.cs` existed in source but Boot.unity had no CaptureGate component, so the capture gate would have produced zero frames — the exact silent failure it exists to catch). Regression-guard pattern: an EditMode scene-presence assert that loads the scene and requires the component (`CaptureGateSceneTests`); binary scenes can't be GUID-grepped, so the EditMode test is the only authoritative reader. Any "this component must exist in the shipped scene" contract needs one.
- **Two scene-build sub-paths — BOTH now bake into the shipped `Boot.unity` (corrected 2026-06-13).** `BootstrapProject.Run` builds ONE `Boot.unity` in two passes: `BuildBootScene()` → `MovementCameraScene.Author` (player / camera / input + the castaway, held axe, hair, craft-spot, beach debris) at `BootstrapProject.cs:53`, THEN `WorldBootstrap.BuildEnvironment()` (Zone-D terrain / scatter / **OCEAN** / lighting / fog / post / skybox — which reaches `LowPolyZoneGen.BuildZone` → `BuildWaterEdge`) at `:62`, then `SaveScene(scene, Boot.unity)` at `:63` re-saves the combined scene. **(HISTORICAL — why the earlier doc said the opposite:** an earlier state had `WorldBootstrap` as a SEPARATE path NOT re-saved into Boot, so the beach shipped with no ocean despite WorldBootstrap holding a water plane (Uma PR #28); U5 / PR #32 fixed it by calling `BuildEnvironment()` + re-saving into Boot.) So world content reaches the soak whether built on the `MovementCameraScene` player path OR the `WorldBootstrap` environment path — but know which path a system lives on (player / props / craft → MovementCameraScene; terrain / ocean / scatter → WorldBootstrap → LowPolyZoneGen). Either way, gate "it's in the shipped Boot.unity" with an EditMode scene-presence test (binary scenes can't be GUID-grepped).
- **"Is what I'm seeing actually the asset?" — the magenta cross-build trace (Drew PR #38, 2026-06-13).** Before tuning a surface's COLOR, prove the surface is even visible. Force the suspect mesh's material to a sentinel color (magenta), rebuild the SHIPPED exe, and pixel-diff the sentinel build against the normal build. If ~0 pixels change (Drew measured 248/921600 = 0.027%, and the suspect "sea" band was byte-IDENTICAL between builds), the mesh contributes no visible pixels — what you're seeing is something ELSE (here: horizon FOG/sky, with the water mesh rendering 0 px). **The magenta-diff tells you the mesh ISN'T VISIBLE, not WHY** — diagnose the cause separately and don't assume occlusion: Drew first hypothesized occlusion (raised terrain in front), but a follow-up `-seaWaterOnly` probe (hide every OTHER mesh) STILL showed 0 water px, proving the real cause was **backface-culling from inverted grid winding** (see Low-poly mesh patterns § −Z grid), not occlusion. A color/material tweak can't fix a not-rendering problem — the magenta trace tells you you're chasing a symptom; the isolate-the-mesh probe tells you the actual cause.
- **Visibility gates need a FIXED-ORBIT capture, not a subject-fit close-up (Devon PR #39, 2026-06-13).** A verify-capture that frames from the settled subject's own bounds (zoom-to-fit) renders the subject at a fixed apparent size REGARDLESS of how large it actually is in the real scene — so it cannot validate gameplay-SCALE visibility. `-verifyAxe`'s zoom-to-fit close-up went green while the held axe was a 0.43u sliver = ~3.7% of the real 14u/55° orbit frame (invisible in gameplay). Any "is X visible to the player" gate must capture from a camera matching the real gameplay distance/FOV (a fixed-orbit framing), NOT a subject-fit close-up. (Third instance of the false-green class, after the no-post verify cam and the stale-SMR-bounds framing — a capture that auto-zooms to its subject is the tell.)
- **Verify a UX-visible feature from the DEFAULT GAMEPLAY CAMERA ANGLE the player actually sees it from — and beware an AVERAGING guard that masks a localized defect (hair-spike, 2026-06-13).** A castaway hair crown-spike passed BOTH a green EditMode geometry guard AND a `-verifyHair` shipped capture, yet the Sponsor still saw it in his soak — because (a) the capture framed the FRONT at a horizon tilt (pitch 20) while the spike sits on the TOP-crown, only visible from the default over-the-shoulder / top-down orbit the player actually uses (front shots: no spike; top-down: spike), and (b) the guard asserted the top-15-ring AVERAGE spread (<0.05u), which a single proud vertex doesn't move enough to trip. Lessons: (1) choose the verify camera to MATCH where the player sees the feature — the default gameplay orbit pitch/azimuth — and shoot MULTIPLE angles (front + top-down + behind), not one convenient angle; (2) make the guard catch a LOCAL maximum (no single vertex protrudes above the surface), not an average that a localized defect can hide under. 4th instance of the false-green class — angle-mismatch + averaging-mask, distinct from no-post / zoom-to-fit / stale-bounds. **RESOLUTION (Devon soakfix5, 2026-06-14):** the persistent hair "tuft" was NOT a proud crown vertex at all — the crown plateau was genuinely flat (the plateau guard correctly passed). It was the **front FRINGE jutting FORWARD** past the face axis (frontmost local z ~0.22u beyond the nominal radius), which projects ABOVE the brow from an over-shoulder down-looking camera + catches the key light → reads as a bright tuft. A world-Y top-vertex / crown-plateau guard is BLIND to forward-jutting geometry — you need a separate guard that the front fringe sits BELOW the crown (`MessyHairCap_FrontFringe_SitsBelowCrown_NoForwardTuft`). **Lesson: a silhouette defect at the TOP of a head can come from FORWARD geometry (the fringe), not just UPWARD geometry — guard both axes.**
- **Guard the PERCEPT, not a proxy geometry metric — a guard satisfiable while the gameplay-cam view still looks wrong is a false-confidence generator (hair false-green wave, 2026-06-14 / pending integration PR soakfix6).** Every false-green in this section shares a root: the guard asserted a geometric predicate (crown-apex spread, front-fringe world-Z, a top-ring average) that a still-wrong-looking mesh could satisfy. The castaway hair false-greened ~5 iterations because each fix flattened the hair toward a dome AND the guards ENFORCED that flatness — but a flat dome reads as a HELMET from the default over-shoulder gameplay cam, exactly the Sponsor's complaint. **When the defect is perceptual ("reads as a dome / a spike / a helmet"), the guard predicate must measure a perceptual proxy — e.g. protrusion count + silhouette roughness sampled from the actual gameplay camera ray, asserting the shape READS as intended — not a geometry-space statistic (vertex spread, bounds ratio, average height) that can diverge from the camera-space silhouette and stay silent.** Test for it: "could a wrong-looking mesh pass this guard?" — if yes, the predicate is a proxy; replace it. *(Specific silhouette-roughness instrument `HairSilhouetteTrace` / guard `TuftedHair_ReadsClumped` are on the unmerged branch `devon/castaway-soakfix6` — defer API cites until that PR merges.)*
- **A custom URP skybox shader drawing OVER geometry mimics a fog/ambient/grade wash — the `-flatSky` isolation probe is the tell (Drew PR #48, world-look; pending merge).** A custom shader assigned to `RenderSettings.skybox` MUST use standard skybox-pass render state (`Cull Off` / `ZWrite Off`, object-space direction, normal clip transform). Forcing the clip depth via `positionCS.xyww` on a Background-queue SubShader makes the gradient render as a FULL-SCREEN FILL over ALL geometry → a uniformly-washed frame that looks exactly like blown fog / ambient / color-grade. Same false-symptom family as the magenta / cull-back classes (the visible symptom points at the wrong system; isolate to find the real cause). Fix: revert to standard skybox-pass render state.
- **Distant geometry reading as "floating translucent shards" is COMPOUNDING FADES, not a transparency/alpha or winding bug (Drew PR #48, 2026-06-15; pending merge).** Far faceted mountains read as ghostly, see-through shards hovering at the horizon — the consultant first hypothesized a winding / surface-alpha bug, but a trace REFUTED it (shader opaque, alpha=1, outward-winding guard already green). - **Sea/water reading as "elevated" or a "floating slab" over the beach is usually a COMPOSITION problem (the seaward view is mostly skybox), NOT a water-Y bug (Drew PR #48 diag `17450d5`, 2026-06-15).** A Sponsor "the water sits above the beach / floats" report trace-REFUTED a water-Y bug: water-Y (`-0.20`) was geometrically correct (below the `+0.05` beach; the sand dips under it at the waterline worldZ -2.0). Asset-fine / view-is-the-problem family. **DOMINANT cause, in the end (Drew PR #48 `3ad8f44`):** oversized **vista landmass islands DRAPING over the play space + occluding the foreground terrain** (it read as a pale/washed "void" frame) — a `-hideVista` isolation proved the terrain renders fine sand/grass once they clear. So ALSO check that distant vista / landmass geometry isn't draping over or burying the play area (add a placement-clearance guard like `Vista_LandmassBases_DoNotDrapeOverThePlaySpace`), not just the sea extent. (The diagnosis chain overturned 3×: water-Y → seaward-composition → terrain-shader → vista-islands-draping — isolate each layer.)
- **A dark LINE / RING across the world is the URP real-time SHADOW-DISTANCE boundary — not a tree shadow, terrain seam, leftover slab, or vertex-color band (Drew PR #50, 2026-06-16).** The Sponsor saw a long straight dark streak across the grass that read "world-fixed" (a similar band from every camera angle). (Both the orchestrator AND the Sponsor first misread it as a tree shadow.)
- **Flat geometry (water) reading as the SKY/skybox color = BACKFACE-CULLED — and a vertex-NORMAL guard does NOT catch it, because URP `Cull Back` culls by triangle WINDING, not the normal (Drew PR #50, 2026-06-16).** The sea read identical to the sky (no horizon); the ticket blamed fog. `-waterProbe` REFUTED fog: the camera ray HIT the water mesh (insideBounds, renderer enabled) yet sampled **0 water px = exactly the `SkyHorizon` color**, and a straight-DOWN capture over open water ALSO showed 0 water px → not occlusion, not fog → the sea's TOP was the BACK face (culled), so the camera saw the skybox THROUGH it. ## Build stripping & shaders

- **Custom shaders must be registered in `AlwaysIncludedShaders`** (Project Settings → Graphics) or the build strips them (spike: vertex-color terrain shader rendered editor-only until registered).
- **URP/Lit ignores vertex color.** Vertex-colored lit geometry (low-poly terrain ramps) needs a custom URP shader.
- **URP package emits `Terrain Standard 4 Layers URP` shader-dependency warnings on first import** — package-internal noise, not project errors; CI "zero console errors" gates must allowlist them (U1 bootstrap finding).
- **SRP-Batcher compliance is a FPS-regression gate, not a style choice — audit it BEFORE merge, not at the Sponsor soak (ticket 86ca9a3b3, Erik §D Evidence D1).** Two things silently drop a renderer out of the SRP batch and cost FPS with NO test catching it: (a) a shader property declared OUTSIDE `CBUFFER_START(UnityPerMaterial)`, and (b) a live `MaterialPropertyBlock`/`SetPropertyBlock` on the renderer (an MPB also disqualifies GPU Resident Drawer — unity6-mastery §2 — and is mutually exclusive with GPU Instancing). The `team/TESTING_BAR.md` visual-UX gate now requires a Frame-Debugger / SRP-Batcher-stats check on any new shader/renderer before the Self-Test Report. **Audited state of the two existing surfaces (86ca9a3b3, both COMPLIANT — no fix filed):**
  - **`LowPolyVertexColor.shader` — ALL properties inside the cbuffer.** The full declared `Properties` set is `_Tint` (Color), `_WaveAmp` (Float), `_WaveLen` (Float), `_WaveSpeed` (Float), `_FogCap` (Range 0-1) — and the `CBUFFER_START(UnityPerMaterial)` block (shader lines 60-66) declares exactly `float4 _Tint; float _WaveAmp; float _WaveLen; float _WaveSpeed; float _FogCap;`. Every property is inside; nothing is outside. (`_LightDirection` in the ShadowCaster pass is a global, NOT a per-material property — correct as-is.) SRP-batchable.
  - **No live `MaterialPropertyBlock` anywhere in the shipped scatter/procedural path.** Grepping `MaterialPropertyBlock|SetPropertyBlock|GetPropertyBlock` across the whole `Assets/Scripts` tree (runtime + editor + tests asmdefs) returns ZERO hits. `LowPolyZoneGen` (the procedural world builder, in the `FarHorizon.Editor` asmdef — it bakes serialized MeshRenderers into Boot.unity at bootstrap, not at runtime) colours every terrain/scatter/water renderer via `mr.sharedMaterial = <inline Material>` (e.g. lines 272/725/775), with distinct quantized-palette material instances NOT persisted as `.mat` assets. That is the SRP-Batcher-friendly path by construction — distinct material instances batch by shader variant (unity6-mastery §2 "proliferate material instances freely"), no MPB break. Keep it MPB-free.
- **Opaque-queue water is the FPS-protecting CHOICE — Transparent-queue water is a deferred upgrade, NOT a missed win (ticket 86ca9a3b3 / Drew PR #48, Erik §D Evidence D2 + Application).** `LowPolyVertexColor.shader` renders water on the **Opaque** queue (`Tags { "RenderType"="Opaque" "Queue"="Geometry" }`, line 41) and separates sea-from-sky via the in-shader `_FogCap` floor (lines 28-37, 131-146), NOT via alpha blending. Moving water to the Transparent queue to get depth-fade foam would (1) **break URP fog composition** — transparent surfaces don't compose with the Exp² distance fog the way opaque ones do, re-opening the sea↔sky horizon problem `_FogCap` was built to solve — and (2) **incur full overdraw on the ~600u ocean extent** (every transparent water pixel re-shaded over whatever's behind it), a real desktop-FPS cost on a large sea plane. So depth-fade foam (which wants the Transparent queue) is a DELIBERATE later PR with its own perf budget, not an oversight in the current opaque water. Don't "upgrade" the water to Transparent casually — it trades a measured FPS win + working fog for a foam nicety.

## FBX / rigs / characters

- **Held props must be posed HAND-LOCAL, never WORLD, or the rotation won't survive a turn.** Setting a held
  prop's `transform.rotation` (world) after parenting it to a hand bone back-solves a localRotation against the
  bone's AUTHORING pose — but a per-frame re-apply of a fixed WORLD rotation (e.g. a nudge tool) pins the prop
  to one world heading. POSITION-following (offset-from-hand each frame) still tracks, so the prop bobs with the
  hand but its rotation stays frozen — when the character re-faces during click-move the prop "points the same
  way on the x axis all the time" (soakfix8 86ca8ce6y, the Sponsor's exact words). FIX: pose the prop with a
  stable `localPosition` + `localRotation` as a CHILD of the bone so BOTH position and rotation ride the bone via
  the hierarchy in every facing — no per-frame world re-apply. The regression guard is a PlayMode test asserting
  `Inverse(hand.rotation) * prop.rotation` is INVARIANT across facings (the bug = it drifted); plus an EditMode
  guard that the prop is a DIRECT child of the bone with a non-identity local rotation.

- **`avatarSetup` T-pose trap:** imported humanoids need the avatar configured from the FBX's own T-pose or retargeted animation mangles limbs.
- **Quaternius Animated-Men FBX clip names carry the `HumanArmature|` prefix** — exact-name clip lookups silently match zero clips (T-pose-mid-walk symptom); use `.Contains` matching plus a `looped < expected` error guard.
- **Normalize intrinsic import height on EVERY character-mesh swap** — meshes vary wildly (UAL mannequin ~1.0u vs Animated-Men ~4.96u); measure bounds and normalize `globalScale` to ~1u before any size system applies, or the camera/shadow/size calibration breaks.
- **Recolor must enumerate ALL materials** — the Animated-Men mesh actually carries SEVEN material slots in import order `[0]Skin [1]Eyes [2]Hair [3]Shirt [4]Pants [5]Socks [6]Shoes`; slot 6 (Shoes) has ZERO faces (the castaway is barefoot) so it's a free slot to repurpose (e.g. a black "Pupil" material paired with a white "Eyes" sclera for cartoon eyes). - **Attaching a prop to a bone on a height-normalized FBX → beware runaway lossy bone scale (86ca8ce6y / Devon PR #29).** The chibi FBX normalizes the MESH down to game scale while its bone transforms carry a large compensating scale — `RightHand_010` had a ~267× `lossyScale`. A naive `SetParent` + "normal" local scale (≈0.18) shipped a 30–50u GIANT axe (caught only in the SHIPPED capture — it framed the whole world; the editor looked plausible). Land a believable prop with a TINY hand-local scale (the axe needed ~`0.0015`), or divide your intended world-scale by the bone's `lossyScale`. **Resolve the attach bone from the `SkinnedMeshRenderer.bones` array by name** (`RightHand_010`), NOT a `transform.Find`/name-scan of the hierarchy — the imported rig carries trap nodes a scan matches first (a mesh-GROUP node named `head` at the model origin vs the real `Head_05` bone; a `RightHand.Dummy_011` sibling vs `RightHand_010`).
- **Held props on an imported-rig bone must be posed in WORLD space, not bone-local (86ca8ce6y / Devon PR #39, 2026-06-13).** Beyond the lossy-scale trap: the imported bone's LOCAL axes are arbitrarily rotated — `RightHand_010`'s local +Y maps to world ≈`(0.48, −0.84, 0.23)` (mostly DOWN). Bone-local offsets are unreliable on these rigs; world-space posing is deterministic.
- **The Mini Chibi Kid is SIX skinned objects sharing ONE atlas, with DEGENERATE skin data — map atlas cells to body regions by ENUMERATING ALL SMRs + per-object UVs, never a single-SMR Y-band / bone-weight split (Devon SOAKFIX4, 2026-06-13).** The chibi avatar is not one `SkinnedMeshRenderer` — it's SIX (e.g. `Object_44` etc.), all sampling the same `mini_material_baseColor.png`. - **Walk-float saga (9 iterations) — RESOLVED at PR #47 `9c5cacd` (CI-green + orch-verified gameplay-cam captures; final confirmation = the Sponsor's over-shoulder soak). TWO real bugs + four hard verification lessons:**
  - **Bug A — the actual float-while-walking symptom: the Mixamo WALK clip authors the HIPS ~0.66u HIGHER than the IDLE clip** (scale-immune `ClipBaselineDiagnose`: Idle sole root-relative `−0.003`, Walk `+0.63..+0.69`) → the rendered body floats ONLY while walking even when the root is correctly grounded.   - **Bug B — why every MEASUREMENT lied (the 8 false-greens before A was found): the Mixamo/Hyper3D FBX bakes a 100× cm→m scale on the `SkinnedMeshRenderer`'s OWN transform** (`model` node `localScale=(100,100,100)`; verts authored in metres). ## Low-poly mesh patterns

- **Welded verts + `RecalculateNormals` = the smooth-shaded look** (averaged vertex normals over coarse facets). Import equivalents: Normals=Calculate with smoothing angle ~60°, not 0°.
- **Thin double-sided foliage: never reuse verts with opposite winding** — coincident opposite-facing triangles average their normals to ~zero → near-black "shards" (iter8 grass bug). Use distinct verts per face AND **up-bias the normals on both faces** (foliage reads as overhead-lit, not as a solid). A normal-distribution probe (count verts with `N·L < threshold` against the key light) is the load-bearing verification — the first plausible fix still shaded dark.
- **Per-instance color-jittered materials cause asset churn** — quantize to a coarse palette and keep materials inline (not `CreateAsset`) so they serialize into the scene.
- **The coarse 12-step palette quantizer splits NEAR-NEUTRAL warm tints into a pink `R>G=B` cast** *(pending integ/soakfix7 merge — `QuantizeFine`, drew/rocks-sourced @ `d358176`, ticket 86ca8m5zu).* The coarse quantizer (above) snaps each channel to 12 steps — fine for saturated colours, but a near-grey warm ramp (R≈G≈B, low chroma) snaps to `(0.667, 0.583, 0.583)` = R>G=B = unmistakable PINK. Diagnostic: a `TINTDIAG` material dump (the mesh/UV are fine; the delivered `_Color` is wrong). Fix: a finer 24-step `QuantizeFine` for low-chroma / near-grey props (preserves R≥G≥B; ~4 mats vs the 1 that pinked). Rule of thumb: if a prop's intended tint has all channels within ~0.1 (chroma < ~0.1), use the fine quantizer; saturated props stay safe on the coarse step.
- **A −Z procedural grid that reuses a +Z grid's triangle winding ships INVERTED normals → backface-culled / invisible (Drew PR #38, 2026-06-13).** The sea grid lays its rows near→far in DECREASING world Z (the sea runs AWAY from the +Z shore) but reused the terrain grid's triangle index order (written for INCREASING Z). Same index order + mirrored vertex layout = faces wound the opposite way → −Y normals → the default URP `Cull Back` culls them from any above-looking camera. The water was NEVER occluded — it was backface-culled (proven: a `-seaWaterOnly` probe that hides every other mesh STILL rendered 0 sea px; the magenta-diff showed 0→55,103 px / 5.98% after the fix). Fix = reverse the triangle winding for the −Z grid (`LowPolyZoneGen.BuildWaterEdge`). Guard = assert every face normal points the expected way (`WaterFacesUpTests`: every `Water_Play` normal·+Y > 0) — a normal-direction guard that runs in EditMode with NO build, the silhouette/normal class. Same family as the foliage opposite-winding bug above — whenever you reuse a grid's index buffer for a grid laid out in a different direction, re-derive or flip the winding.
- **ANY procedural FLAT-SHADED mesh with explicit per-face normals must enforce OUTWARD winding or URP `Cull Back` silently culls it — same class as the −Z grid bug above** *(pending integ/soakfix7 merge — `FacetedRock`, drew/rocks-sourced @ `d358176`, ticket 86ca8m5zu).* Flat shading needs unwelded verts so each face carries its own baked normal; faces wound INWARD → −normals → culled from every camera → the mesh renders as a thin sliver (or nothing) despite being geometrically chunky. **Diagnostic to tell "culled" from "actually-thin":** a geometry-dump probe (`ROCKDIAG`) reporting per-face extent ratios — ratio ~0.9 + 0 thin faces means the mesh IS chunky, so invisibility is culling, not geometry. Fix: force every generated face normal outward + flip winding; guard `FacetedRock_AllFacesPointOutward` (EditMode, no build). Pattern: when a builder bakes per-face normals explicitly (instead of `RecalculateNormals`, which self-corrects outward), verify winding independently — and note the EDITOR may still draw the culled mesh, masking the runtime cull until the shipped build.

## CI architecture — merge-gate vs advisory PlayMode (PR #80 `86caapwmt`, 2026-06-18)

- **The self-hosted runner has a CONFIRMED play-mode-ENTER env-deadlock: headless Unity freezes "Entering Playmode" and never self-quits.** It is ENVIRONMENTAL + fixture-presence-triggered — it fires BEFORE any single fixture runs, so there is no one bad fixture to `[Ignore]`; EditMode is unaffected (308/308 green) and the SAME PlayMode pattern passed earlier on green main run `27685445841`. NOT a code bug.
- **Why a step-level `timeout-minutes` + `continue-on-error` on an in-line PlayMode step does NOT keep the job green (proven on run `27769261766`):** the hung headless Unity IGNORES the GitHub step-cancel that `timeout-minutes` fires, so the orphaned `Unity.exe` keeps the runner pinned until the runner **LOSES COMMUNICATION with the server** — an INFRASTRUCTURE failure (`The self-hosted runner lost communication with the server`) that `continue-on-error` CANNOT suppress (it catches step exit codes, not a lost runner). The in-line PlayMode step killed the WHOLE `unity` job — Build + capture never ran — turning the merge gate red on an env problem.
- **The structural fix — split the workflow into a reliable blocking gate + an advisory PlayMode job:**
  - **Merge gate = `structure` (hosted) + `unity` (self-hosted): bootstrap → EditMode → BuildWindows → shipped-build capture.** All three Unity gates are reliable; all BLOCKING. The `unity` job no longer runs PlayMode at all.
  - **`playmode` = a SEPARATE, ADVISORY, NON-REQUIRED, FAIL-FAST job.** `continue-on-error: true` at the JOB level → it can NEVER block a merge (do NOT add its check to required status checks). `timeout-minutes: 20` is a HARD cap so a FUTURE play-mode-enter deadlock fails FAST + NAMED (the `playmode` job goes red, clearly attributable) instead of hanging to the 90-min job cap and crashing the runner. `needs: unity` SERIALIZES it AFTER the merge-gate build so the two Unity invocations never contend the single runner's PackageCache (EPERM — `[[single-unity-build-slot-serializes-orchestration]]`). `if: always()` lets it run for signal even when the merge-gate job failed.
- **Re-folding PlayMode into the blocking gate** (once the env-deadlock root cause is fixed) is tracked on ticket `86cab7u42`. Until then: PlayMode runs for SIGNAL but is never load-bearing for merge.
- **Reviewer note:** a red `playmode` check on a PR is NOT a merge blocker by itself — read the `unity` (merge-gate) + `structure` checks for the gate verdict, and check the `playmode` job's tail to distinguish a real PlayMode test failure (worth investigating) from the env-deadlock timeout (expected until `86cab7u42`).
